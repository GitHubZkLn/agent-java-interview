一、用户上下文传递（ThreadLocal专题）

Q1: 用户登录后，根据Cookie解析出用户信息，如何让其他业务Service都能拿到这个信息，且在多线程情况下子线程也能拿到，要求对业务代码无入侵？
参考答案：
核心方案：ThreadLocal + 请求上下文封装

一、基本实现（单线程/同步场景）：
1. 创建用户上下文持有类
// 创建全局上下文类 public class UserContext { private static final ThreadLocal<User> CURRENT_USER = new ThreadLocal<>(); public static void setUser(User user) { CURRENT_USER.set(user); } public static User getUser() { return CURRENT_USER.get(); } public static void clear() { CURRENT_USER.remove(); // 重要！防止内存泄漏 } }
2. 登录时解析Cookie并设置
// 登录服务中设置用户上下文 public class LoginService { public void login(String cookie) { User user = parseCookieAndGetUser(cookie); UserContext.setUser(user); // 设置到ThreadLocal } }
3. 业务代码直接获取，无需传参
// 业务代码直接获取 public class OrderService { public void createOrder() { User user = UserContext.getUser(); // 直接获取，无需参数传递 // 业务逻辑... } }
二、异步场景解决方案（阿里TransmittableThreadLocal）：
1. 引入依赖，使用TTL替代ThreadLocal
<!-- Maven依赖 --> <dependency> <groupId>com.alibaba</groupId> <artifactId>transmittable-thread-local</artifactId> <version>2.14.5</version> </dependency>
2. 使用TTL封装上下文
// 使用TransmittableThreadLocal替代ThreadLocal public class UserContext { private static final TransmittableThreadLocal<User> CURRENT_USER = new TransmittableThreadLocal<>(); public static void setUser(User user) { CURRENT_USER.set(user); } public static User getUser() { return CURRENT_USER.get(); } public static void clear() { CURRENT_USER.remove(); } }
3. 包装线程池（推荐方式）
// 方式一：包装线程池 @Configuration public class TtlConfig { @Bean public ExecutorService executorService() { return TtlExecutors.getTtlExecutorService( Executors.newFixedThreadPool(10) ); } } // 方式二：包装单个任务 @Test public void testAsync() { Runnable task = TtlRunnable.get(() -> { User user = UserContext.getUser(); // 自动继承父线程的值 // 业务逻辑... }); executor.submit(task); }
追问：为什么ThreadLocal.remove()必须在请求结束时调用？
原因分析：
  • ThreadLocalMap的Entry继承WeakReference<ThreadLocal>，但value是强引用
  • 如果Thread使用线程池，线程会被复用
  • 请求结束后如果没调用remove()，该线程再次被复用时仍持有旧数据
  • 可能造成数据串台（A用户数据泄露给B用户）
最佳实践：Filter中统一清理
@WebFilter(urlPatterns = "/*") public class UserContextFilter implements Filter { @Override public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain) throws IOException, ServletException { try { chain.doFilter(req, resp); } finally { UserContext.clear(); // 必须finally中清理 } } }
追问：InheritableThreadLocal和TransmittableThreadLocal有什么区别？
特性InheritableThreadLocalTransmittableThreadLocal
父子线程传递支持支持
线程池传递❌ 不支持✅ 支持
线程复用数据错乱数据正确传递
使用方式直接使用需包装线程池或任务
来源JDK内置阿里开源
关键区别：InheritableThreadLocal只在创建子线程时传递一次值，但线程池中的线程是复用的，不存在真正的父子关系，所以InheritableThreadLocal无效。
Q2: ThreadLocal底层原理是什么？为什么能实现线程隔离？
参考答案：
ThreadLocal底层结构：
  • 每个Thread对象内部有一个ThreadLocalMap（本质是Entry数组）
  • ThreadLocal作为Key（弱引用），Value作为值
  • ThreadLocalMap使用线性探测法解决哈希冲突
ThreadLocal核心源码结构
// 核心get方法 public T get() { Thread t = Thread.currentThread(); ThreadLocalMap map = getMap(t); // 获取当前线程的ThreadLocalMap if (map != null) { ThreadLocalMap.Entry e = map.getEntry(this); if (e != null) return (T)e.value; } return initialValue(); // 首次调用返回初始值 } // ThreadLocalMap的Entry实现 static class Entry extends WeakReference<ThreadLocal<?>> { Object value; Entry(ThreadLocal<?> k, Object v) { super(k); value = v; } }
为什么线程隔离：每个线程有独立的ThreadLocalMap实例，ThreadLocal对象作为Key存入各自线程的Map中，所以互不影响。

二、Spring Boot 启动后初始化代码的方式

Q3: SpringBoot启动后，还想直接初始化自动执行某个代码，Spring提供了哪些方式？
参考答案：
Spring提供了5种主要方式，按执行时机排序：
方式执行时机优先级特点
@PostConstructBean创建后，依赖注入完成时⭐⭐⭐ 最先Java EE标准注解
InitializingBean.afterPropertiesSet()@PostConstruct之后⭐⭐Spring接口，紧耦合
@Bean(initMethod="...")InitializingBean之后⭐配置解耦，适合第三方Bean
ApplicationRunner容器完全启动后按@Order顺序可解析启动参数
CommandLineRunnerApplicationRunner之后按@Order顺序获取原始命令行参数
方式1：@PostConstruct
@Component public class DataInitializer { @PostConstruct public void init() { System.out.println("@PostConstruct 执行：初始化数据"); } }
方式2：InitializingBean
@Component public class ConfigInitializer implements InitializingBean { @Override public void afterPropertiesSet() throws Exception { System.out.println("InitializingBean 执行：配置加载"); } }
方式3：@Bean initMethod
@Configuration public class BeanConfig { @Bean(initMethod = "init") public MyBean myBean() { return new MyBean(); } }
方式4：ApplicationRunner（推荐）
@Component @Order(1) // 数字越小优先级越高 public class MyApplicationRunner implements ApplicationRunner { @Override public void run(ApplicationArguments args) throws Exception { System.out.println("ApplicationRunner 执行"); // args可以解析 --key=value 格式参数 System.out.println("非选项参数: " + args.getNonOptionArgs()); System.out.println("选项参数: " + args.getOptionNames()); } }
方式5：CommandLineRunner
@Component @Order(2) public class MyCommandLineRunner implements CommandLineRunner { @Override public void run(String... args) throws Exception { System.out.println("CommandLineRunner 执行"); System.out.println("原始参数: " + Arrays.toString(args)); } }
追问：为什么@PostConstruct比InitializingBean先执行？
根本原因：BeanPostProcessor的执行顺序

Spring Bean初始化流程：
  1. 构造器执行
  2. BeanPostProcessor.postProcessBeforeInitialization
  3. @PostConstruct处理（由CommonAnnotationBeanPostProcessor执行）
  4. InitializingBean.afterPropertiesSet()
  5. @Bean(initMethod)
  6. BeanPostProcessor.postProcessAfterInitialization
@PostConstruct由CommonAnnotationBeanPostProcessor在postProcessBeforeInitialization中处理，而InitializingBean的调用在更后面。
追问：ApplicationRunner和CommandLineRunner有什么区别？
对比ApplicationRunnerCommandLineRunner
参数类型ApplicationArguments（封装对象）String[]（原始数组）
参数解析✅ 支持解析 --name=value❌ 只能拿到原始字符串
获取值方式args.getOptionNames()Arrays.toString(args)
使用场景需要解析启动参数时简单场景或不需要参数时
执行顺序：ApplicationRunner先执行 → CommandLineRunner后执行

三、大文本内容存储与检索方案

Q4: 在业务工具中出现了很多大文本内容，我们一般怎么存储？存储在哪里？如何实现高效检索？
参考答案：
一、存储方案选择（根据数据规模和业务需求）：
存储方案适用场景优点缺点
MySQL TEXT中等规模（<1MB/条）事务支持、简单SQL查询大文本检索效率低
MySQL + 全文索引小规模全文检索无需额外组件性能有限，不适合海量
MongoDB非结构化文档存储Schema灵活、文档存储不支持复杂关联查询
Elasticsearch海量全文检索倒排索引、高性能需要额外组件
OSS/OSS + 数据库索引超大文件存储成本低、CDN加速不支持直接检索
MySQL存储方案
-- TEXT类型选择 CREATE TABLE article ( id BIGINT PRIMARY KEY, title VARCHAR(255), content LONGTEXT, -- 最多4GB create_time DATETIME ); -- 添加全文索引（MySQL 5.6+ InnoDB支持） ALTER TABLE article ADD FULLTEXT INDEX ft_content (title, content); -- 查询语法 SELECT * FROM article WHERE MATCH(title, content) AGAINST('关键词' IN NATURAL LANGUAGE MODE);
Elasticsearch存储+检索（推荐方案）
// 1. 定义Document @Document(indexName = "articles") public class ArticleDocument { @Id private String id; @Field(type = FieldType.Text, analyzer = "ik_max_word") private String title; @Field(type = FieldType.Text, analyzer = "ik_max_word") private String content; } // 2. 存储文档 @Autowired private ElasticsearchRestTemplate template; public void saveArticle(Article article) { ArticleDocument doc = convertToDocument(article); template.save(doc); } // 3. 全文检索 public List<ArticleDocument> search(String keyword) { NativeSearchQuery query = new NativeSearchQueryBuilder() .withQuery(QueryBuilders.matchQuery("content", keyword)) .build(); return template.search(query, ArticleDocument.class) .getSearchHits().stream() .map(SearchHit::getContent) .collect(Collectors.toList()); }
追问：MySQL的LIKE模糊查询和ES的全文检索有什么区别？
对比维度MySQL LIKEElasticsearch全文检索
原理逐行匹配，无法利用索引倒排索引，提前分词建索引
性能全表扫描，数据量越大越慢毫秒级响应，支持海量数据
分词不支持中文分词IK等中文分词器支持
相关性排序不支持TF-IDF、BM25相关性评分
并发能力受数据库连接限制分布式架构，高并发
结论：数据量<100万条可用MySQL全文索引，>100万条推荐ES。
追问：OSS存储大文本如何实现元数据检索？
OSS + 数据库元数据方案
// 1. 文本存OSS，元数据存MySQL CREATE TABLE file_metadata ( id BIGINT PRIMARY KEY, file_name VARCHAR(255), oss_url VARCHAR(512), -- OSS存储路径 file_size BIGINT, -- 文件大小 content_hash VARCHAR(64), -- 内容哈希（去重用） create_time DATETIME ); // 2. 上传文件到OSS public String uploadToOss(String content) { String key = "docs/" + UUID.randomUUID() + ".txt"; ossClient.putObject(bucketName, key, new ByteArrayInputStream(content.getBytes())); return key; } // 3. 元数据检索（通过MySQL） public List<FileMetadata> searchByName(String keyword) { return jdbcTemplate.query( "SELECT * FROM file_metadata WHERE file_name LIKE ?", "%" + keyword + "%" ); } // 4. 需要全文时从OSS下载 public String getContent(String ossKey) { OSSObject obj = ossClient.getObject(bucketName, ossKey); return new String(obj.getObjectContent().readAllBytes()); }

四、Spring 注入方式冲突解决方案

Q5: Spring中有个UserService接口，有多个实现类，注入时该选哪一个？怎么解决注入冲突？
参考答案：
问题背景：当一个接口有多个实现类时，Spring无法确定使用哪个Bean，会报错：
NoUniqueBeanDefinitionException
场景：UserService有多个实现
public interface UserService { User getUserById(Long id); } @Service public class UserServiceImplA implements UserService { } @Service public class UserServiceImplB implements UserService { }
方案1：@Primary指定默认实现
@Service @Primary // 标记为首选Bean public class UserServiceImplA implements UserService { } // 注入点无需改动，Spring会自动选择A @Autowired private UserService userService; // 注入的是A
方案2：@Qualifier指定Bean名称
@Autowired @Qualifier("userServiceImplB") // 指定Bean名称 private UserService userService; // 注入的是B
方案3：@Resource按名称注入
@Resource(name = "userServiceImplB") private UserService userService;
方案4：自定义Bean名称
@Service("vipUserService") public class UserServiceImplVip implements UserService { } @Autowired @Qualifier("vipUserService") private UserService userService;
方案5：注入所有实现类到集合
// 如果业务需要遍历所有实现 @Autowired private List<UserService> allUserServices; // 注入所有实现 @Autowired private Map<String, UserService> userServiceMap; // key是Bean名称
追问：@Primary和@Qualifier同时使用，谁的优先级更高？
@Qualifier优先级更高，会覆盖@Primary的选择。
@Service @Primary public class UserServiceImplA implements UserService { } @Service public class UserServiceImplB implements UserService { } // 使用@Qualifier明确指定，会忽略@Primary @Autowired @Qualifier("userServiceImplB") // 实际注入B，而非@Primary的A private UserService userService;
追问：@Autowired和@Resource有什么区别？
对比维度@Autowired@Resource
来源Spring注解JSR-250标准注解（Java提供）
默认注入方式byTypebyName
required属性支持（默认true）不支持
构造函数注入支持部分支持
参数名匹配需要额外配置自动按参数名匹配

五、过滤器、拦截器与AOP的区别

Q6: 在Spring中，过滤器(Filter)、拦截器(Interceptor)、AOP有什么区别？各自用在什么场景？
参考答案：
一、执行层次与顺序：
请求到达 → Filter链 → DispatcherServlet → Interceptor.preHandle() → Controller → Interceptor.postHandle() → 视图渲染 → Interceptor.afterCompletion() → 响应客户端
二、核心区别对比：
对比维度Filter（过滤器）Interceptor（拦截器）AOP（切面）
规范归属Servlet/Jakarta EESpring FrameworkSpring Framework
执行位置Web容器层Spring MVC层Spring Bean层
触发时机进入容器后，Servlet前DispatcherServlet后方法调用前后
拦截范围所有请求（含静态资源）仅Controller接口方法级别
Bean访问需FilterChain传递可注入任意Bean可注入任意Bean
Filter：适合"全局外围"处理
@WebFilter(urlPatterns = "/*") public class CharacterEncodingFilter implements Filter { @Override public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain) { req.setCharacterEncoding("UTF-8"); chain.doFilter(req, resp); } } // 典型场景： // - 字符编码设置 // - CORS跨域处理 // - XSS/CSRF防护 // - 全链路日志追踪
Interceptor：适合"业务入口"校验
@Component public class LoginInterceptor implements HandlerInterceptor { @Override public boolean preHandle(HttpServletRequest req, HttpServletResponse resp, Object handler) { String token = req.getHeader("token"); if (token == null) { resp.setStatus(401); return false; } return true; } } // 注册方式 @Configuration public class WebMvcConfig implements WebMvcConfigurer { @Override public void addInterceptors(InterceptorRegistry registry) { registry.addInterceptor(new LoginInterceptor()) .addPathPatterns("/api/**") .excludePathPatterns("/api/login"); } } // 典型场景： // - 登录验证 // - 权限校验 // - 请求日志
AOP：适合"业务方法"增强
@Aspect @Component @Slf4j public class PerformanceAspect { @Around("execution(* com.example..*Service.*(..))") public Object logTime(ProceedingJoinPoint point) throws Throwable { long start = System.currentTimeMillis(); Object result = point.proceed(); log.info("{} 耗时: {}ms", point.getSignature(), System.currentTimeMillis() - start); return result; } } // 典型场景： // - 方法性能监控 // - 事务管理 // - 日志记录 // - 重试机制
追问：Filter和Interceptor的异常处理有什么区别？
场景FilterInterceptor
异常捕获需在Filter中try-catch异常会被DispatcherServlet统一处理
返回错误响应必须手动编写可配合@ControllerAdvice统一处理

6.1 微服务相关

Q7: 如何设计一个不停机迁移数据库的方案？
参考答案：
核心方案：双写 + 灰度切换

  4. 阶段4：全量切换
    • 新库全量读
    • 旧库下线

Q8: 分布式事务如何保证一致性？Seata的AT模式原理是什么？
参考答案：
常见分布式事务解决方案：
方案原理优点缺点
2PC协调者先prepare再commit强一致阻塞长，存在数据不一致风险
TCCTry-Confirm-Cancel三阶段性能好业务侵入大
Saga正向编排+补偿操作长事务支持补偿逻辑复杂
AT（Seata）自动补偿，无代码侵入低侵入需要undo_log表

Seata AT模式原理：
  1. TM（事务发起者）开启全局事务，获取XID
  2. RM（分支事务）执行业务SQL
  3. RM注册分支事务，记录undo_log
  4. TM提交/回滚全局事务
  5. Seata Server通知所有RM：提交则删除undo_log，回滚则执行undo_sql恢复数据

6.2 Redis进阶

Q9: Redis的BigKey问题是什么？如何发现和解决？
参考答案：
BigKey定义：单个String类型value > 10KB，或Hash/List/Set/ZSet元素数量过多。

问题影响：内存不均匀、阻塞主线程、网络带宽占用大、集群迁移困难
发现BigKey
# 命令行发现 redis-cli --bigkeys # 扫描大Key redis-cli -h host -p 6379 --scan | head -10000 | \ xargs -L 1 redis-cli -h host -p 6379 DEBUG OBJECT KEYNAME
解决方案：
  • 拆分：大Hash拆分为多个小Hash
  • 压缩：String类型用压缩算法（zlib）
  • 异步删除：UNLINK代替DEL（非阻塞删除）
  • 数据归档：历史数据迁移到MySQL
Q10: Redis主从同步原理？全量同步和增量同步分别什么时候触发？
参考答案：
同步流程：
  1. 全量同步（RDB快照）：
    • 触发条件：slave首次连接 / slave数据太旧（repl_backlog不足）
    • master生成RDB文件，发送给slave
    • slave清空本地数据，加载RDB
  2. 增量同步（命令传播）：
    • 全量同步后，master实时发送写命令
    • 通过repl_backlog环形缓冲区记录增量命令
    • slave持续同步offset

Q12: JDK 21 Scoped Values是什么？和ThreadLocal有什么区别？
参考答案：
Scoped Values（JEP 446）：JDK 21正式特性，用于替代ThreadLocal的更安全方案。
使用示例
public class Server { // 不可变绑定，只能写入一次 private static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance(); public static User currentUser() { return CURRENT_USER.get(); } public void handle(Request req) { // 在这个lambda中可以通过CURRENT_USER.get()获取值 ScopedValue.runWhere(CURRENT_USER, authenticate(req), () -> { processRequest(); }); } }
Scoped Values优势：
  • 值不可继承（更安全）
  • 作用域结束时值自动失效（无需手动清理）
  • 不可变绑定，防止意外修改

七、综合场景设计题

Q13: 如何设计一个接口调用链追踪系统？要求记录每个调用的耗时和调用关系。
参考答案：
设计方案：TraceID串联 + Filter/Interceptor记录
1. Trace上下文
public class TraceContext { private static final ThreadLocal<TraceInfo> CONTEXT = new ThreadLocal<>(); public static void set(String traceId) { CONTEXT.set(new TraceInfo(traceId, System.currentTimeMillis())); } public static TraceInfo get() { return CONTEXT.get(); } public static void clear() { CONTEXT.remove(); } }
2. TraceFilter（生成/获取TraceID）
@WebFilter("/*") public class TraceFilter { @Override public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain) { HttpServletRequest httpReq = (HttpServletRequest) req; String traceId = httpReq.getHeader("X-Trace-ID"); if (traceId == null) { traceId = UUID.randomUUID().toString().replace("-", ""); } TraceContext.set(traceId); HttpServletResponse httpResp = (HttpServletResponse) resp; httpResp.setHeader("X-Trace-ID", traceId); try { chain.doFilter(req, resp); } finally { TraceContext.clear(); } } }
3. TraceInterceptor（记录方法耗时）

Q19: MVCC是如何实现的？ReadView是什么？RR和RC隔离级别下ReadView有什么区别？

Q20: MySQL主从同步原理是什么？如何解决主从延迟问题？
参考答案：
主从复制原理：

  2. Dump线程：从库IO线程连接主库，请求Binlog
  3. 传输Binlog：主库Binlog Dump线程发送Binlog给从库
  4. Relay Log：从库IO线程写入中继日志

异步复制低（可能丢失事务）高
半同步复制中（至少一个从库确认）中
全同步复制高（所有从库确认）低

主从延迟解决方案：读从库、强制走主库、并行复制、缩短Binlog、业务优化。

消费者Rebalance：消费者组内成员变化时，重新分配分区的过程。

  • 使用静态成员：Consumer配置group.instance.id，消费者重启不触发Rebalance

// 前端：分片切割 function uploadFile(file) { const chunkSize = 2 * 1024 * 1024; // 2MB const chunks = Math.ceil(file.size / chunkSize); const fileHash = await calculateMD5(file); // 秒传验证用 // 查询已上传的分片 const uploadedChunks = await api.getUploadedChunks(fileHash); // 并发上传未上传的分片 const tasks = []; for (let i = 0; i < chunks; i++) { if (!uploadedChunks.includes(i)) { const start = i * chunkSize; const end = Math.min(start + chunkSize, file.size); const chunk = file.slice(start, end); tasks.push(uploadChunk(chunk, i, fileHash)); } } await Promise.all(tasks); // 通知服务端合并 await api.mergeChunks(fileHash, chunks); }
