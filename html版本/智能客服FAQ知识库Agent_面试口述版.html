Q1：为什么 FAQ 知识库检索结果差？有什么优化方案？RAG

**分三层说（对应根因分析框架）：**

**① 文档与分块层**
表格 / PDF 解析不完整，关键信息被截断 → 换解析器或加元数据字段兜底；
固定字数切块把问答对拆散 → 改用**语义分块**（按段落 / 标题 / QA 对切），加 **Chunk Overlap** 防止边界割裂；
Embedding 模型与业务语义不匹配 → 换微调 embedding 或用域内 embedding。

**② 检索策略层**
单向量检索对缩写词、人名、精确 ID 召回差 → 加 **BM25 / 关键词检索**，走**混合检索（Hybrid Search）**；
Query 与知识库表述不一致 → 加 **Query 改写**（同义扩展、多 query）、**HyDE**（先生成假设答案再检索）；
召回太泛或太窄 → 调 Top-K 与相似度阈值；先粗召再精召。

**③ 重排序与生成层**
大量召回噪声 → 加 **Rerank**（Cross-Encoder 二轮打分选出 Top-3）；
去除重复片段（余弦相似度 > 0.87 去重）、页眉页脚等噪声修剪；
参考代码里的链路正是：`get_similar` → `rerank` → `chat` 三步。

**口述收口：** 「先定位是『召回不足』还是『排序不准』，再从数据、策略、模型三层对症下药。实战中从 50%～60% 提升到 90%+，核心靠的是**混合检索 + Rerank**。」
# 参考代码链路（来自面试截图右侧片段） def get_similar(user_query): # 语义向量检索，召回 Top-K return vector_db.search(embedding(user_query), top_k=20) def rerank(results): # 二轮精排，去噪声、选出 Top-3 return cross_encoder.rerank(results, top_n=3) def chat(context, user_query): # 精排结果 + 用户问题 → LLM 生成答案 return llm.generate(system_prompt(context) + user_query)

Q2：如何保证客服回答的质量？Agent

**四维回答（结合智能客服架构实践）：**

**① 召回质量：** 检索结果 Top-N 与用户问题高度相关，从源头控制幻觉——上一题的混合检索 + Rerank 链路解决的就是这个问题。

**② Prompt 约束：** System prompt 里明确「只基于检索结果回答」「不知道就说不知道」「格式要求（引用来源）」「禁止编造」——越清晰的指令，模型越不容易跑偏。

**③ 置信度机制：** 检索得分低于阈值时**直接转人工**，不强行生成；生成前判断是否有效上下文；出现拒绝回答时记录日志供分析。

**④ 评测与反馈闭环：** 上线后持续收集「踩」反馈，分析高频 badcase；定期抽样人工评估（相关性、准确性、有害性）；关键指标：命中率、拒答率、用户满意度。生产上还要监控 **P99 延迟**，防止慢查询拖垮体验。

**口述一句：** 「质量不是靠 LLM 自己保证的，是从**检索 → Prompt → 置信度 → 反馈**全链路一起控制的。」

Q3：想快速集成大模型客服，选什么方案？快在哪里？工程

**推荐思路（按真实项目调整）：**

**技术栈：RAG 框架 + LLM API**
上层用 LangChain / LlamaIndex / Qwen-Agent 等封装好 RAG 链路；
Embedding 用 sentence-transformers（离线）或 OpenAI Embedding；
向量库用 ChromaDB / FAISS（零配置）；
LLM 用 OpenAI GPT-4 / Claude API / 通义等。

**快速集成体现（三点必说）：**
**无需训练**：RAG 不需要微调，上传文档即可生效，知识更新小时级；
**工具链成熟**：Embedding + 向量库 + LLM 三行代码串联，开源 demo 当天可跑；
**接入成本低**：HTTP API 暴露 QA 接口，前端 / APP / 飞书机器人当天可对接。

**口述：** 「我们第一版从零到跑通用了 3 天，核心是选对了 RAG 框架自己填业务逻辑，而不是从零写检索。生产上再加监控、限流、AB 分流。」

Q6：Agent 开发最重要的是什么？Agent

**核心答案（结合业界共识）：**

**① 工具边界设计：** 工具定义是否清晰、Schema 是否稳定，直接决定 Agent 能否稳定调用。工具描述模糊、参数边界不清晰是翻车最多的地方。

**② 循环终止条件：** 最大步数、重复检测（检测连续调用同一工具同参数）、置信度 / 预算 / 超时熔断，防死循环是生产级别的必修课。

**③ 上下文质量：** 「垃圾进垃圾出」，检索质量 > 模型能力上限；RAG 链路不稳，生成再强也是无根之木。

**④ 评测体系：** 可衡量的质量指标（命中率、幻觉率、响应时长、badcase 闭环）比「感觉效果不错」更重要。

**⑤ 可观测：** Tracing、工具调用日志、每步输入输出，方便排查 Badcase。

**口述一句：** 「Agent 的上限是大模型，但**下限是工程**：工具、边界、可观测、评测，这几样没做好，模型再强也会翻车。」

Q7：介绍一下你做的最好的项目？开放

**STAR 结构（建议 2 分钟）：**

**S（背景）：** 业务痛点是什么，客服每天处理多少工单，人力成本多少，为什么要上 AI。

**T（目标）：** 解决什么问题，设定可量化的目标（如命中率从 X% 提升到 Y%、响应时长从 A 降到 B）。

**A（行动）：** 技术选型（为什么用 RAG 而不是微调）、架构设计（分层、模块职责）、你负责哪一段（检索 / 生成 / 评测 / 工程）、踩过的坑和怎么修的。

**R（结果）：** 量化指标（召回率、准确率、用户满意度、节省人力），上线后数据变化。

**加分项：**
提到「踩过的坑」（Embedding 不匹配、分块导致召回差、幻觉问题及解决方案）；
提到「迭代过程」（第一版什么样，第二版改了什么、为什么改）；
提到「可扩展性」（加新知识库、多语言支持、多轮对话）。

Q8：对客服行业的看法？开放

**三段话结构（展现思考深度）：**

**第一段：行业趋势。** 「客服正在从成本中心变成**数据资产中心**。好的客服不只是解决问题，还在收集用户真实需求、发现产品问题——这些数据反过来驱动产品迭代。」

**第二段：AI 赋能点。** 「80% 的 FAQ 类问题 AI 可以秒级回答；剩下 20% 复杂问题转人工时，AI 可以帮人工快速总结上下文、推荐答案，让人工效率翻倍，同时也降低培训成本。」

**第三段：挑战与机会。** 「最大的挑战是**知识库维护成本**和**答案质量把控**（幻觉、时效性）；机会在于多模态（语音 / 视频）、主动服务（预测用户问题）和全链路数据分析。」

**口述一句：** 「我不把客服只当『应答』，而是当成用户生命周期里最重要的触点之一——它直接影响用户信任和复购。」
