Q1：== 与 equals 的区别是什么？hashCode 有什么关系？

**== 是比较运算符，行为取决于比较对象的类型：**
· 基本类型（byte/short/int/long/float/double/boolean/char）：比较实际存储的值是否相等。
· 引用类型：比较两个引用指向的内存地址，即是否为同一个对象。

**equals() 是 Object 类的实例方法：**
· 默认实现与 == 完全一致（比较地址）。
· String、Integer 等常用类重写了 equals()，改为比较对象的内容是否相等。

**hashCode() 返回 int 哈希值：**
· 主要用于 HashMap、HashSet 等哈希表的快速定位。
· 核心契约：若 x.equals(y) == true，则 x.hashCode() 必须等于 y.hashCode()。
· 反过来不成立：hashCode 相同，equals 不一定为 true（哈希冲突）。

**为什么要同时重写？**
若只重写 equals 而不重写 hashCode，会导致相等的两个对象被插入 HashMap 的不同位置，查找语义被破坏。同样，重写 hashCode 后也要同步重写 equals。
追问：String 的 equals 是如何重写的？new String("a") 和 "a" 的 == 结果是什么？Integer 在 [-128, 127] 范围内 == 比较结果如何？

Q2：String、StringBuilder、StringBuffer 的区别？String 为什么不可变？

**核心区别——可变性：**
· String：底层是 final char[]（JDK9+ 改为 final byte[]），不可变对象，每次"修改"都生成新对象。
· StringBuilder / StringBuffer：继承自 AbstractStringBuilder，底层是可变 char[]，直接在数组上修改。

**线程安全性：**
· StringBuffer：对所有关键方法加 synchronized，线程安全，性能较低。
· StringBuilder：不加锁，线程不安全，性能更高。
· 日常推荐：没有线程安全需求时优先用 StringBuilder。

**String 不可变的好处：**
① 支持字符串常量池，节省内存；② hashCode 只计算一次，HashMap 查询效率高；③ 多线程安全，无需同步；④ 防止子类重写，保护核心行为；⑤ 安全性（数据库账号密码等不可被篡改）。

**字符串常量池：** 字面量如 "abc" 存入常量池；new String("abc") 在堆中创建额外对象。
intern()：JDK6 将字符串拷贝到常量池；JDK7+ 将堆中的引用放入常量池。
追问：String s = new String("abc") 创建了几个对象？String s1 = "a" + "b" + "c" 呢？编译后一样吗？

Q3：final、static、volatile、transient 关键字的作用？

**final——限制可变性：**
· 修饰变量：基本类型值不可变，引用类型引用地址不可变（对象内部仍可变）。必须显式初始化。
· 修饰方法：方法不可被重写（但可被重载）。
· 修饰类：类不可被继承，如 String。

**static——类成员共享：**
· 修饰变量：类变量，所有实例共享同一份，内存中只有一份副本。
· 修饰方法：可直接 类名.方法() 调用，不能访问实例变量/方法。
· 修饰代码块：静态代码块，类加载时执行一次，用于初始化静态资源。

**volatile——可见性与禁止重排序（多线程相关）：**
· 保证可见性：线程修改后，其他 CPU 核心的缓存立即失效，强制从主内存重新读取。
· 禁止指令重排序：通过内存屏障阻止编译器和 CPU 的重排序优化。
· 不保证原子性：count++ 这种复合操作仍需额外同步。
· 应用场景：状态标记量、DCL 单例模式中的 instance 变量。

**transient——跳过序列化：** 修饰变量，反序列化时跳过该字段，初始化为默认值。静态变量天然不参与序列化。

Q4：面向对象三大特性是什么？封装、继承、多态如何理解？

**① 封装（Encapsulation）——数据安全**
将对象的状态（属性）和行为（方法）捆绑在一起，通过访问修饰符隐藏内部实现细节。
· private 属性 + public getter/setter：控制读写权限。
· 好处：提高可维护性，降低耦合度，防止非法修改数据。

**② 继承（Inheritance）——代码复用**
子类复用父类的属性和方法，用 extends 关键字。Java 只支持单继承，但支持多层继承。
· 子类继承父类所有 non-private 成员，通过 super.xxx() 调用父类实现。
· 子类构造函数必须调用父类构造函数（显式或隐式）。

**③ 多态（Polymorphism）——接口统一，行为多样**
同一操作作用于不同对象，产生不同结果。必要条件：继承 + 重写 + 父类引用指向子类对象。
· 编译时多态（重载 Overload）：同名方法不同参数，编译时静态绑定。
· 运行时多态（重写 Override）：父类引用在运行时动态绑定到子类实现（虚方法表 vtable 分派）。
· 好处：可替换性、可扩充性、接口性、灵活性。

Q6：重载（Overload）与重写（Override）的区别？

**重载（Overload）——编译时多态，同类不同参**
在同一类中定义多个方法名相同但参数列表不同（类型、数量或顺序）的方法。
· 编译器根据参数签名静态绑定调用版本。
· 返回类型可以不同，但不能仅以返回值区分（编译器无法区分）。
· 可改变访问修饰符和异常声明。

**重写（Override）——运行时多态，父子类方法签名一致**
发生在父子类之间，子类用不同实现覆盖父类方法。
· 方法名、参数列表、返回类型必须完全相同（JDK5+ 支持协变返回类型）。
· 访问修饰符不能比父类更严格。
· 不能重写被 final、static、private 修饰的方法。
· 运行时通过虚方法表（vtable）动态分派。

维度重载 Overload重写 Override
存在范围同一类中父类与子类之间
参数列表必须不同必须完全相同
绑定时机编译时静态运行时动态
多态关联无关核心体现

Q7：Java 是值传递还是引用传递？参数传递的内存模型是怎样的？

**核心结论：Java 只有值传递（Pass by Value），没有引用传递。** 这是理解 Java 参数机制的关键。

**基本类型的值传递：**
方法接收的是值的完整副本。方法内的修改完全不影响原变量。
int a = 5; change(a); → a 仍为 5。
原因：基本类型存储在栈中，每个方法调用有独立的栈帧。

**引用类型的传递（容易混淆）：**
传递的是引用地址的副本，而不是引用本身。
· 修改对象属性（如 p.name = "Alice"）：影响原对象，因为引用副本仍指向同一个堆对象。
· 重新赋值引用（如 p = new Person()）：不影响原引用，因为只改变了副本的指向。

**String 和包装类型的特殊性：**
String 是不可变类，任何"修改"都创建新对象，表现如同值传递。
String name = "原始"; modify(name); → name 仍为"原始"。

操作类型是否影响原变量原因
修改对象属性✅ 影响引用副本指向同一堆对象
重新赋值引用❌ 不影响只改变副本的指向
基本类型❌ 不影响纯值副本
String / 包装类❌ 不影响不可变或值传递
追问：为什么 Java 不设计成引用传递？如果要修改引用本身怎么办？

Q8：ArrayList 和 HashMap 的底层原理？JDK1.7 和 1.8 的核心区别？

**ArrayList 底层原理——动态数组：**
· 底层使用 Object[]，默认初始容量 10。
· 扩容机制：容量不足时扩容为原来的 1.5 倍。
· 末尾添加：O(1)；指定位置插入/删除：O(n)（需移动元素）；随机访问：O(1)。
· 线程不安全。

**HashMap 底层原理——JDK1.7 vs 1.8：**
特性JDK 1.7JDK 1.8+
数据结构数组 + 链表数组 + 链表 + 红黑树
插入方式头插法尾插法
死循环风险高（并发扩容头插导致环链）低（尾插避免）
链表转红黑树无链表长度 ≥ 8 且 数组长度 ≥ 64
查询复杂度最好 O(1)，最差 O(n)最好 O(1)，红黑树 O(log n)

**关键参数：** 默认初始容量 16，负载因子 0.75，扩容阈值 = capacity × loadFactor。
**寻址算法：**index = (n - 1) & hash，JDK1.8 对 hash 做二次扰动（高16位异或低16位）减少碰撞。
**扩容优化：** JDK1.8 扩容时，元素要么在原索引，要么在原索引 + oldCapacity，无需重新计算 hash。
追问：HashMap 为什么选择红黑树而不是 AVL 树？负载因子 0.75 是怎么来的？线程安全吗？

Q9：常用数据结构有哪些？各有什么使用场景？List / Set / Map / Queue 对比？

**四大接口对比：**
类型元素有序性核心实现适用场景
List可重复有序ArrayList / LinkedList / Vector有序列表、按索引访问
Set唯一无序（部分有序）HashSet / LinkedHashSet / TreeSet去重、集合运算
Map键值对无序（部分有序）HashMap / LinkedHashMap / TreeMap键值映射、快速查找
Queue可重复FIFO 或优先级LinkedList / PriorityQueue / ArrayDeque任务调度、异步处理

**List 选择：**
· ArrayList：频繁随机访问（O(1)），尾插性能好。推荐默认选择。
· LinkedList：频繁在任意位置插入/删除（O(1)），但随机访问 O(n)。
· Vector：已过时，需要线程安全时用 Collections.synchronizedList() 或 CopyOnWriteArrayList。

**Set 选择：**
· HashSet：只需去重，无序，高效。
· LinkedHashSet：去重 + 保持插入顺序。
· TreeSet：去重 + 自然排序或自定义排序（底层红黑树）。

**Map 选择：**
· HashMap：键值高效查找，无序。
· LinkedHashMap：键值查找 + 保持插入顺序或访问顺序（LRU Cache）。
· TreeMap：键值查找 + 按键排序（范围查询）。

**Queue / Deque：**
· ArrayDeque：高性能双端队列，不支持 null，不支持并发。
· PriorityQueue：二叉堆实现，支持优先级排序（非 FIFO），默认小顶堆。
· LinkedList：既实现了 List 又实现了 Queue，支持 null。

Q10：Java 内部类有哪几种？静态内部类和成员内部类的区别？

**四种内部类对比：**
类型定义位置持有外部引用访问外部成员能否定义静态成员内存泄漏风险
成员内部类外部类成员位置✅ 有（隐式 this$0）所有成员（含私有）❌ 不能有

**成员内部类：** 定义在外部类的成员位置，无 static 修饰。持有外部类隐式引用，可访问外部类所有成员。创建需先创建外部类对象：new Outer().new Inner()。

**静态内部类：** 不持有外部类引用，独立于外部类对象存在。只能访问外部类静态成员。创建不需外部类对象：new Outer.NestedClass()。
· 应用场景：Builder 模式（ public static class Builder）、单例模式（Holder 方式）、逻辑分组。

**局部内部类和匿名内部类：** 只能访问 final 或事实上 final 的局部变量（原因：方法结束后局部变量销毁，但内部类对象可能仍存活，Java 通过复制值解决）。
· 匿名内部类没有类名，编译后以 OuterClass$1 命名。广泛用于事件监听、回调、一次性接口实现。

Q11：Java 泛型是什么？类型擦除是什么？通配符 ? extends T 和 ? super T 的区别？

**泛型的价值：** 在类型安全与代码复用间取得平衡，避免强制类型转换错误。

**类型擦除（Type Erasure）：**
Java 泛型信息只存在于编译时，编译后所有泛型类型被擦除为原始类型。例如 List<String> 和 List<Integer> 编译后都变为 List。这是为了保持向后兼容性，减少虚拟机改动。
· 无法创建泛型数组（如 T[] arr = new T[10]）。解决方案：反射或 Array.newInstance(Class<T>, size)。
· 无法用 instanceof 判断泛型类型（运行时类型信息丢失）。

**通配符对比：**
类型语法能否读取能否写入应用场景
无界通配符List<?>可（Object）❌ 不能只读操作
上界通配符List<? extends T>可（T）❌ 不能生产者（读取数据）
下界通配符List<? super T>可（Object）可（T）消费者（写入数据）

**PECS 原则（Producer-Extends, Consumer-Super）：**
· 集合作为生产者（提供数据）用 ? extends T。
· 集合作为消费者（接收数据）用 ? super T。
· 经典案例：Collections.copy(List<? super T> dest, List<? extends T> src)，dest 是消费者，src 是生产者。

**注意：**List<Integer> 不是 List<Number> 的子类型，需用通配符来解决子类型问题。

Q12：Java 异常处理机制？throw、throws、try-catch-finally 的区别？

**异常体系：**
· Error：JVM 层面的严重问题（如 OutOfMemoryError），程序无法处理，不应捕获。
· Exception：分为 受检异常（Checked）（编译期必须处理，如 IOException）和 非受检异常（Unchecked）（运行时异常，如 NullPointerException）。

**throw vs throws：**
· throw：在方法体内手动抛出异常对象，如 throw new RuntimeException("参数错误")。
· throws：在方法签名后声明可能抛出的异常类型，将异常上报给调用者。

**try-catch-finally 规则：**
· try 必须有；catch 和 finally 至少出现一个。
· finally 不一定执行：① try 之前已抛异常；② 执行了 System.exit()；③ finally 中抛出异常。
· catch 中有 return，finally 仍会执行。基本类型返回值不受 finally 影响（值已压栈）；引用类型若 finally 修改了引用则有影响。若 finally 也有 return，则覆盖 catch 的返回值。
· 建议使用try-with-resources（Java 7+），自动关闭 AutoCloseable 资源。

**finalize() 已被废弃：** Object.finalize() 由垃圾回收器调用，不应依赖它释放资源。
追问：受检异常和非受检异常的设计考量是什么？Java 为什么选择不支持多异常捕获（如 catch(A | B e)）？

Q13：强引用、软引用、弱引用、虚引用有什么区别？各自的使用场景？

**四种引用强度从强到弱：**
引用类型回收时机能否通过 get() 获取典型使用场景
强引用永不回收（内存不足时抛 OOM）可以常规对象引用 Object o = new Object()
软引用内存不足时回收可以内存敏感缓存（如图片缓存）
弱引用每次 GC 时必回收可以防止内存泄漏（如 ThreadLocalMap.Entry）
虚引用随时可能被回收永远不行跟踪对象回收（堆外内存管理）

**强引用：** 最普遍的引用方式。JVM 宁愿抛出 OutOfMemoryError 也不会回收强引用对象。

**软引用：** 描述非必需对象。在系统将要发生内存溢出之前会被回收。适合做内存敏感的高速缓存，内存充足时保留，不足时自动回收。

**弱引用：** 被弱引用关联的对象只能生存到下一次垃圾收集发生之前。无论内存是否足够，GC 扫描到时都会立即回收。
· 典型案例：ThreadLocal 的 Entry 继承 WeakReference<ThreadLocal<?>>，当 ThreadLocal 对象无强引用时，下一次 GC 时 key 被回收，避免内存泄漏。

**虚引用（幻影引用）：** 最弱的引用，不影响对象生命周期。唯一作用是在对象被回收时收到系统通知。必须与 ReferenceQueue 配合使用。
· 典型案例：NIO DirectByteBuffer 借助虚引用 + ReferenceQueue 管理堆外内存（off-heap memory）。
追问：WeakHashMap 的实现原理是什么？ReferenceQueue 在软引用和虚引用中起什么作用？

Q14：Java 8 有哪些新特性？Lambda、Stream、Optional 的核心用法？

**① Lambda 表达式：**
将函数作为方法参数传递。基本语法：(参数) -> 函数体。
示例：list.forEach(e -> System.out.println(e))、(a, b) -> a + b。
Lambda 可捕获局部变量、实例变量和静态变量。

**② 函数式接口：**
仅有一个抽象方法的接口，用 @FunctionalInterface 标记。可包含多个 default/static 方法。
常见函数式接口：Runnable、Callable、Function<T,R>、Predicate<T>、Consumer<T>、Supplier<T>。

**③ Stream API：**
函数式处理集合数据的利器。特点：无存储、不修改源数据、延迟执行、可无限流、元素只访问一次。
· 中间操作：filter / map / flatMap / distinct / sorted / limit / skip（懒加载）。
· 终端操作：collect / reduce / forEach / count / toArray / min / max（触发执行）。
· 示例：list.stream().filter(x -> x > 0).map(x -> x * 2).collect(Collectors.toList())。

**④ Optional 类：**
避免 NPE（空指针异常）的包装类。
· Optional.ofNullable(x).orElse("默认值")
· Optional.ofNullable(x).map(...).orElse(null)
· Optional.ofNullable(x).isPresent() → Optional.ofNullable(x).ifPresent(...)

**⑤ 接口的 default 和 static 方法：**
Java 8 允许接口定义实现，打破了接口不能有实现的历史。使用场景：向后兼容的接口扩展（如 Collection.stream()）。

Q15：Object 类有哪些核心方法？equals、hashCode、clone 如何正确重写？

**Object 类核心方法（共 11 个）：**
toString / equals / hashCode / clone / getClass / notify / notifyAll / wait / finalize / registerNatives / wait(long timeout)

**toString()：** 默认返回 类名@十六进制哈希码（如 Person@15db9742）。建议所有业务类都重写，返回关键属性的可读字符串。

**equals() 重写规范（五大约定）：**
① 自反性：x.equals(x) 为 true。② 对称性：x.equals(y) == y.equals(x)。③ 传递性。④ 一致性（多次调用结果相同）。⑤ 与 null 比较返回 false。
· 标准重写模板：== 自检 → instanceof 类型检查 → 强转 → 字段比较。

**hashCode() 重写原则：**
· 必须与 equals 成对重写，否则哈希表行为异常。
· 推荐使用 IDE 自动生成或 Objects.hash(field1, field2, ...)。
· 乘数常用 31（31 = 2^5 - 1，便于 JVM 优化为 (i << 5) - i）。

**clone()：** 默认是浅拷贝，返回 Object 类型。必须实现 Cloneable 接口，否则抛 CloneNotSupportedException。
· 正确姿势：protected Object clone() { return super.clone(); }。
· 现代推荐：优先使用拷贝构造器或拷贝工厂方法，更直观且可控制深拷贝。

Q16：BIO、NIO、AIO 的区别是什么？为什么 NIO 是主流？

**核心概念区分：**
· 同步 / 异步：关注消息通知机制——异步是操作系统完成 I/O 后主动通知，同步是应用主动等待或轮询结果。
· 阻塞 / 非阻塞：关注线程状态——阻塞是线程被挂起，非阻塞是线程立即返回可执行其他操作。

模型全称线程模型优点缺点适用场景
BIO同步阻塞 I/O一连接一线程逻辑简单直观线程数随连接线性增长，无法支撑高并发连接数少且固定的简单应用
NIO同步非阻塞 I/O一线程处理多连接（Reactor 模式）少量线程管理大量连接，资源利用率高编程模型复杂，有 Selector 空轮询 Bug高并发、长连接、低活跃（聊天 / RPC / Kafka）
AIO异步非阻塞 I/O基于 Proactor 模式，操作系统完成后通知理论上性能最高，无需轮询Linux 上实现不成熟，API 复杂，生态不完善超高并发长连接（Windows IOCP）

**NIO 三大核心组件：**
· Channel（通道）：类似流，但可双向读写（不像流单向）。
· Buffer（缓冲区）：读写数据的容器，支持多种数据类型。
· Selector（选择器）：单线程监听多个 Channel 的 I/O 事件，实现多路复用。

**为什么 NIO 是主流而非 AIO？**
NIO 在 Linux 平台实现成熟（epoll）、生态完善（Netty、gRPC、Kafka 都基于 NIO）。AIO 在 Linux 上依赖 epoll 但未充分优化，实际上是线程池模拟异步。Netty 等主流框架均选择 NIO 架构。
