OpenSpec SDD + Harness Thinking

AI Coding 的质量，不靠“多写点提示词”，靠一条生产级工程流水线。

在存量复杂代码里，AI 生成代码只是中间动作。真正要保证质量，需要先把需求规约化，再把项目知识、内部规范、验证脚本、预发环境、流量回放、灰度发布和复盘沉淀串起来。

OpenSpec 解决“这次要做什么”，Coding Harness 解决“怎么可控地做、怎么证明能上线”。

S
Spec Firstproposal / design / tasks / requirement scenarios
H
Harness Assets业务术语、错误码、上下游、架构边界、验证脚本
G
Quality Gates编译、单测、契约、DB dry-run、安全扫描、CI
P
Production Proof预发、流量回放、影子流量、灰度、指标回滚
01

为什么存量复杂代码需要 Coding Harness？

因为 AI 能写出“语法正确”的代码，但生产系统要的是“业务正确、可验证、可回滚”。

存量系统的难点，不是代码文件多这么简单，而是大量知识藏在历史逻辑、上下游契约、数据口径、错误码、灰度规则和故障复盘里。AI 可以从代码里推断“现在怎么写”，但很难自动知道“为什么必须这么写”。

隐性业务知识

字段单位、状态机、兼容逻辑、配置中心约定、错误码语义、历史禁忌，很多都不在当前需求里。

上下游耦合

一个小接口可能影响前端、客户端、BI、消息消费方、风控、结算系统，单看当前模块很容易误判。

验证链路长

编译和单测只是底线。企业生产还需要契约测试、预发部署、真实流量验证、灰度和自动回滚。

结论：Coding Harness 不是替代开发流程，而是把 AI Coding 放进企业已有的工程质量体系里。
02

阿里和得物的实践，可以归纳成四个共同点

这些实践不是围绕“提示词技巧”，而是围绕“工程约束和反馈闭环”。

阿里：把 AI Coding 变成可编排的工程系统

文章来源：阿里云转载《Harness Engineering：耗时一周，我是如何将应用的AI Coding率提升至90%的》

文章核心不是某个神奇提示词，而是把项目知识、规范和验证流程沉淀成 `.harness` 目录：agent 角色、编码规范、需求规格、业务上下文、自动化脚本和模板。AI 不再是拿到一句需求就开写，而是先进入一个被设计好的项目环境。

.harness/
  agents/        # feature-dev / test-dev / issue-fix 等专业化角色
  standards/     # 编码、接口、DB、日志、事务规范
  context/       # 业务术语、字段含义、状态、上下游
  specs/         # 需求规格和变更说明
  scripts/       # 本项目固定验证命令

阿里：SDD 解决“做什么”，Harness 解决“怎么可控地做”

文章来源：微信公众号《告别“氛围编程”：基于 Harness 治理和 SDD 的团队级 AI 研发范式演进与实践》

这条线强调把“氛围编程”拉回研发流程。很多 AI Coding 翻车并不是模型不会写，而是需求没有结构化：接口字段没定、边界条件没说、验收标准没有、业务例外靠口口相传。SDD 先把需求拆成规格、计划、任务和验收；Harness 再把这些规格接入代码仓库、工具链、测试和 review 流程。

得物：全栈、多仓、可模仿

文章来源：得物技术《基于 Harness + SDD + 多仓管理模式的 AI 全栈开发实践》

得物的全栈实践很务实：不要让 AI 凭空创造，而是让它模仿项目里已经跑通的模块。前端、后端、接口文档、SDD 文档放在同一个 workspace，让 agent 能看到跨仓契约，避免前端字段、后端 DTO、数据库含义各说各话。

得物：Claude Code Harness 的运行时治理

文章来源：得物技术《Claude Code Harness 工程：数仓侧落地方案》

这篇更偏运行时治理：复杂数仓或工程任务里，context compact 后容易丢金额单位、分区写法、SQL 规范等临时约束。它的思路是把这些约束写进项目级说明、memory、hooks、subagents 和 skill 文件里，能用脚本拦截的就不要靠模型记忆。

03

OpenSpec 生成什么？Harness 补什么？

OpenSpec 不是替代 Harness，而是企业 Coding Harness 里的规格层。

层次 产物 主要内容 谁来维护
OpenSpec SDD proposal.md / design.md / tasks.md / spec.md 这次需求为什么做、怎么做、拆成哪些任务、满足哪些 Requirement 和 Scenario。 AI 生成第一版，人 review 业务边界和验收标准。
项目入口 AGENTS.md / CLAUDE.md 项目定位、关键目录、启动命令、测试命令、架构边界、高风险目录。 团队手动沉淀，AI 可辅助整理。
内部知识 .harness/context 业务术语、错误码、状态机、字段单位、上下游契约、历史故障。 领域 owner 维护，AI 根据复盘持续补充。
工程规范 .harness/standards API 兼容、DB migration、MQ 幂等、日志 trace、灰度发布、回滚规范。 架构/平台/测试团队共同维护。
确定性验证 .harness/scripts + CI/CD 编译、单测、契约测试、schema diff、DB dry-run、安全扫描、流量回放、灰度指标门禁。 平台化沉淀，AI 只能调用，不应绕过。
04

企业生产级 Coding Harness 最终实践步骤

从需求到上线后复盘，一条完整可审计、可验证、可回滚的链路。

企业级 Coding Harness 不能只停留在“AI 生成 spec + 跑单测”。更完整的链路应该是：OpenSpec 负责需求规约，Harness 负责项目知识和执行约束，CI/CD 负责确定性门禁，预发/回放/灰度负责真实环境验证，最后把问题反哺回规则和测试。

1

先建长期 Harness 资产，不要每次从聊天开始

企业项目里要先准备一组长期文档和脚本，让 AI 每次进项目都有稳定入口。OpenSpec 管“本次变更”，Harness 管“长期规则”。这部分第一版通常要人工补，后续可以让 AI 根据故障复盘和 review 反馈持续更新。

AGENTS.md / CLAUDE.md
- 项目定位、关键目录、启动命令、测试命令
- 架构边界：哪些层不能互相调用
- 高风险目录：migration、支付、资金、权限、定时任务

.harness/context/
- domain-glossary.md：业务术语、状态、枚举、字段单位
- error-codes.md：错误码含义、可复用错误码、禁止新增规则
- upstream-downstream.md：上下游接口、依赖方、兼容要求
- risk-cases.md：历史事故、易错点、线上禁忌

.harness/standards/
- api-contract.md：接口兼容、字段新增/删除规范
- db-migration.md：DDL、索引、回滚、数据修复规范
- message-idempotency.md：MQ、任务、重试、幂等规范
- observability.md：日志、trace、metrics、告警规范
2

用 OpenSpec 做 SDD：先规约，再编码

需求进来后，不直接让 AI 改代码，而是先执行 `/opsx:propose`。AI 可以读取仓库、已有 specs、AGENTS.md 和 `.harness/context` 生成第一版变更包；人重点 review 业务边界、上下游影响、验收场景和风险点。

/opsx:propose 添加优惠券过期提醒功能

生成：
openspec/changes/add-coupon-expire-reminder/
  proposal.md   # 为什么做、做什么、不做什么、影响范围
  design.md     # 技术方案、数据变更、幂等、失败处理、风险
  tasks.md      # 可执行任务清单，每一步都能验证
  specs/
    coupon-reminder/spec.md
                 # Requirement + Scenario，作为验收标准

人工 review 重点：
- 是否遗漏业务边界
- 是否破坏接口兼容
- 是否涉及错误码、状态机、数据迁移
- 是否需要上下游确认
- 每个 Scenario 是否能变成测试或验证动作
3

让 AI 按 spec 小步实现，并强制找参照模块

进入 `/opsx:apply` 或编码阶段后，要求 AI 先读 `tasks.md`，再找相似模块，最后一项一项实现。不要让一个 agent 一口气改完整条链路；高风险场景可以拆成开发 agent、测试 agent、review agent。

编码阶段约束：
- 每次只处理 tasks.md 中一个小任务
- 实现前必须列出 2-3 个相似模块
- 优先复用现有 service、DTO、错误码、状态机
- 不允许扩大改动范围
- 涉及资金/权限/数据迁移必须等待人工确认

分工方式：
- feature agent：按 spec 实现代码
- test agent：根据 Scenario 补单测/集成测试
- review agent：独立检查 diff、兼容性、风险点
- fix agent：只根据 CI/测试失败做最小修复
4

合并前门禁：不靠“看起来对”，靠确定性检查

AI 写完以后，企业级 Harness 必须接入 PR/CI 门禁。单测和编译只是底线，还要覆盖接口契约、数据库变更、安全、架构边界和可观测性。能脚本化的规则，不要靠 prompt 让模型记住。

PR / CI Gate：
- 编译、类型检查、lint、格式化
- 单元测试、集成测试、E2E 回归
- OpenAPI / RPC schema diff
- API contract test
- DB migration dry-run、回滚脚本检查、慢 SQL explain
- 依赖漏洞扫描、secret 扫描、权限变更检查
- 架构边界检查：禁止 controller 直连 DAO 等
- 日志、trace、metrics 是否补齐
- 覆盖率和关键 Scenario 覆盖检查
5

预发验证：部署一个真实运行环境

复杂存量系统不能只在本地跑测试。合并前后要有预发、临时 namespace 或 preview 环境，连接尽量接近生产的中间件和配置，但使用隔离数据。这里验证的是“服务真的能跑起来，链路真的能走通”。

预发环境验证：
- 部署新版本服务或临时 preview 环境
- 启动健康检查、配置检查、依赖连通性检查
- 跑接口冒烟、核心链路 E2E
- 定时任务、MQ 消费、补偿任务单独验证
- 数据迁移 dry-run 和回滚演练
- 观察启动日志、错误日志、trace、慢 SQL
- 和上一个稳定版本做关键响应对比
6

用真实流量验证：回放、影子流量、Dark Launch

很多边界 case 只存在于真实请求里。生产级 Harness 可以录制线上流量，在预发或隔离环境回放；也可以把线上请求复制一份给新版本，只观察新版本结果，不返回给用户。

真实流量验证：
- Traffic Replay：录制生产 API 请求，在预发环境回放
- Shadow Traffic：线上请求复制给新版本，响应丢弃
- Dark Launch：功能代码上线但不对用户开放，只观察链路
- 新旧版本对比：状态码、响应字段、异常率、耗时、业务结果

适合场景：
- 查询逻辑重构
- 推荐/排序/规则引擎调整
- 优惠券、库存、订单等复杂业务链路
- AI 修改了历史兼容逻辑但人造测试覆盖不全
7

灰度发布：用指标决定是否继续放量

上线不是一次性全量，而是 1% 到 100% 的受控过程。每个阶段都要观察技术指标和业务指标，超过阈值自动暂停或回滚。这里 Harness 的作用，是把“发布是否安全”变成可观测、可决策的门禁。

灰度策略：
- Feature Flag 控制功能开关
- Canary：1% → 5% → 10% → 25% → 50% → 100%
- 按用户、租户、城市、业务线分批放量
- 自动回滚：错误率、P95/P99、CPU、内存、消息积压超阈值

必须观察：
- 技术指标：错误率、延迟、超时、慢 SQL、线程池、MQ 堆积
- 业务指标：下单率、核销率、提醒发送量、投诉量、转化率
- 兼容指标：老客户端错误、上下游失败、错误码分布
8

复盘写回：把一次问题变成长期能力

企业级 Coding Harness 最重要的是闭环。AI 犯错、CI 拦截、预发失败、灰度回滚，都不应该只修代码；还要把问题写回 OpenSpec、Harness 规则、测试、hook 或监控，下一次让系统自动拦住。

反哺动作：
- spec.md：补遗漏的 Requirement / Scenario
- AGENTS.md：补项目级硬规则
- .harness/context：补业务术语、错误码、上下游约定
- .harness/standards：补接口、DB、MQ、日志规范
- scripts/hooks：补确定性检查
- tests：补单测、契约测试、回放 case
- monitoring：补指标、告警、dashboard

判断标准：
同类问题下一次不应该再靠人肉发现。
企业级实践的核心不是“让 AI 多写代码”，而是把 AI 放进一条可审计、可验证、可回滚、可持续学习的研发流水线里。OpenSpec 负责把需求变清楚，Harness 负责让执行不跑偏，生产验证负责证明它真的能上线。
