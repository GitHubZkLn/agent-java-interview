OpenSpec SDD + Harness Thinking

AI Coding 的质量，不靠“多写点提示词”，靠一条生产级工程流水线。

在存量复杂代码里，AI 生成代码只是中间动作。真正要保证质量，需要先把需求规约化，再把项目知识、内部规范、验证脚本、预发环境、流量回放、灰度发布和复盘沉淀串起来。

OpenSpec 解决“这次要做什么”，Coding Harness 解决“怎么可控地做、怎么证明能上线”。

S

H
Harness Assets业务术语、错误码、上下游、架构边界、验证脚本
G

P

01

为什么存量复杂代码需要 Coding Harness？

AI 会写代码，但企业真正要的是：业务不跑偏、边界不破坏、上线可验证。

存量系统的难点，不是代码文件多，而是大量知识藏在历史逻辑、上下游契约、数据口径、错误码、灰度规则和故障复盘里。AI 可以从代码里推断“现在怎么写”，但很难自动知道“为什么必须这么写”。

隐性业务知识

字段单位、状态机、兼容逻辑、配置中心约定、错误码语义、历史禁忌，很多都不在当前需求里。

上下游耦合

一个小接口可能影响前端、客户端、BI、消息消费方、风控、结算系统，单看当前模块很容易误判。

验证链路长

编译和单测只是底线。企业生产还需要契约测试、预发部署、真实流量验证、灰度和自动回滚。

Coding Harness 不是替代开发流程，而是把 AI Coding 放进企业已有的工程质量体系里。
02

四篇技术文章详解：它们到底在讲什么？

这部分不只做结论，按文章脉络拆出问题、方案、落地动作和可借鉴点。

阿里第一篇：从 Prompt / Context 走向 Harness Engineering

文章来源：阿里云转载《Harness Engineering：耗时一周，我是如何将应用的AI Coding率提升至90%的》

这篇文章的核心场景是复杂企业 Java 应用：代码多年迭代、业务链路长、模块边界复杂。作者遇到的问题不是 AI 不会写代码，而是 AI 经常写出“语法正确、风格像样、业务语义却错”的代码。尤其在十万行级别存量系统里，单次 prompt 很难让模型记住所有项目规矩。

它批判的做法

只靠 Prompt Engineering 让 AI “按专家方式回答”，或者只靠 Context Engineering 把更多文件塞给 AI。前者改善表达，后者改善材料，但都不能保证执行过程可控。

它提出的转向

把 AI Coding 当成工程系统来设计：有任务角色、有项目规范、有业务上下文、有验证脚本、有变更记录，有一个能反复复用的 `.harness` 工作区。

典型目录思想

`agents` 放专业化 agent，`standards` 放编码规范，`context` 放业务知识，`specs` 放需求规格，`scripts` 放验证脚本，避免所有规则散落在聊天记录里。

最值得学的点

质量保障不是最后人工扫一眼，而是前置写进 agent 指令和验证脚本里。AI 每次执行都要按项目规矩来，而不是每次从零解释。

.harness/
  agents/        # feature-dev / test-dev / issue-fix 等专业化角色
  standards/     # 编码、接口、DB、日志、事务规范
  context/       # 业务术语、字段含义、状态、上下游
  specs/         # 需求规格和变更说明
  scripts/       # 本项目固定验证命令

阿里第二篇：告别“氛围编程”，用 SDD + Harness 做团队级研发

文章来源：微信公众号《告别“氛围编程”：基于 Harness 治理和 SDD 的团队级 AI 研发范式演进与实践》

这篇更关注团队研发范式。它指出很多团队看起来 AI 出码率很高，但真实提效并没有那么高，因为研发不是“写完代码”就结束，还包括需求理解、方案设计、评审、测试、联调、上线和回滚。没有规约的 AI Coding，本质上容易变成“氛围编程”：看起来热闹，实际上返工很多。

SDD 解决什么

先把需求变成结构化规格：目标、非目标、接口契约、数据变更、边界条件、验收场景。它解决“到底要做什么”。

Harness 解决什么

把规格接入仓库、工具链、测试、review 和发布流程。它解决“怎么让 AI 可控地做”。

团队级价值

让需求、方案、任务、测试和上线不再散在个人对话里，而是变成可以 review、可以追溯、可以沉淀的工程资产。

落地提醒

不要把 SDD 做成好看的文档。每个 Requirement / Scenario 都要能转成测试、验证命令或人工验收点。

阶段没有 SDD/Harness有 SDD/Harness
需求一句自然语言需求直接开写。先生成 proposal / design / spec，人工 review 后再实现。
实现AI 凭经验生成一版代码。按 tasks 小步执行，先找参照模块，再做最小改动。
验证靠人看 diff 觉得差不多。用 CI、契约测试、预发、回放、灰度指标证明质量。

得物第一篇：Harness + SDD + 多仓管理，让 AI 做全栈改造不跑偏

文章来源：得物技术《基于 Harness + SDD + 多仓管理模式的 AI 全栈开发实践》

得物这篇非常贴近日常业务开发。它不是让 AI 自己“创造一套新设计”，而是让 AI 模仿项目里已经跑通的模块。全栈任务最容易出问题的是前后端契约不一致、字段理解不一致、复用率低、review 成本高，所以它强调多仓 workspace、明确参照文件和分阶段联调。

多仓工作区

把前端、后端、接口文档、SDD 文档放在同一个 workspace，让 AI 能看到跨仓契约，避免前端字段、后端 DTO、数据库含义各说各话。

让 AI 模仿

明确告诉 AI 参考哪个页面、哪个接口、哪个 service、哪个 DTO。存量系统里，“像项目自己人写的”比“看起来高级”重要。

先问清设计问题

比如主键、分页、逻辑删除、排序、优先级自增、接口错误码、是否需要兼容老数据，这些问题要在编码前暴露。

分阶段验证

先 mock，再联接口，再端到端验证。前后端可以并行，但必须用同一份 spec 和验收标准对齐。

得物第二篇：Claude Code Harness 在数仓侧的运行时治理

文章来源：得物技术《Claude Code Harness 工程：数仓侧落地方案》

这篇更偏 Claude Code 的运行时治理。数仓任务的难点不是只写 SQL，而是上下游血缘、分区规范、金额单位、调度依赖、历史口径和自测 SQL 都非常重要。context compact 后，AI 可能丢掉临时约束，所以文章强调把规则放到项目级说明、memory、hooks、subagents 和 skill 文件里。

项目级上下文

把长期约束写进 `CLAUDE.md` 或项目说明，像金额单位、分区字段、表命名、禁止写法这种规则不能只靠临时对话。

Hooks 自动拦截

能用脚本检查的规则就做成 hooks，例如 SQL 规范、危险 DDL、分区缺失、禁止全表扫描等。

Subagents 分工

把高 token 的血缘分析、SQL review、自测生成拆给独立 subagent，避免主上下文被噪声挤满。

Skill 文件沉淀

把领域流程、常见校验、自测模板和历史经验沉淀成 skill，让 AI 按流程调用，而不是每次重新解释。

03

OpenSpec 生成什么？Harness 补什么？

OpenSpec 是规格层，Harness 是执行和治理层。两者结合，才像企业级 AI Coding。

层次产物主要内容谁来维护
OpenSpec SDDproposal.md / design.md / tasks.md / spec.md这次需求为什么做、怎么做、拆成哪些任务、满足哪些 Requirement 和 Scenario。AI 生成第一版，人 review 业务边界和验收标准。
项目入口AGENTS.md / CLAUDE.md项目定位、关键目录、启动命令、测试命令、架构边界、高风险目录。团队手动沉淀，AI 可辅助整理。
内部知识.harness/context业务术语、错误码、状态机、字段单位、上下游契约、历史故障。领域 owner 维护，AI 根据复盘持续补充。
工程规范.harness/standardsAPI 兼容、DB migration、MQ 幂等、日志 trace、灰度发布、回滚规范。架构/平台/测试团队共同维护。
确定性验证.harness/scripts + CI/CD编译、单测、契约测试、schema diff、DB dry-run、安全扫描、流量回放、灰度指标门禁。平台化沉淀，AI 只能调用，不应绕过。
04

企业生产级 Coding Harness 最终实践步骤

从需求到上线后复盘，一条完整可审计、可验证、可回滚的链路。

企业级 Coding Harness 不能只停留在“AI 生成 spec + 跑单测”。更完整的链路应该是：OpenSpec 负责需求规约，Harness 负责项目知识和执行约束，CI/CD 负责确定性门禁，预发/回放/灰度负责真实环境验证，最后把问题反哺回规则和测试。

1

先建长期 Harness 资产，不要每次从聊天开始

企业项目里要先准备一组长期文档和脚本，让 AI 每次进项目都有稳定入口。OpenSpec 管“本次变更”，Harness 管“长期规则”。这部分第一版通常要人工补，后续可以让 AI 根据故障复盘和 review 反馈持续更新。

AGENTS.md / CLAUDE.md
- 项目定位、关键目录、启动命令、测试命令
- 架构边界：哪些层不能互相调用
- 高风险目录：migration、支付、资金、权限、定时任务

.harness/context/
- domain-glossary.md：业务术语、状态、枚举、字段单位
- error-codes.md：错误码含义、可复用错误码、禁止新增规则
- upstream-downstream.md：上下游接口、依赖方、兼容要求
- risk-cases.md：历史事故、易错点、线上禁忌
2

用 OpenSpec 做 SDD：先规约，再编码

需求进来后，不直接让 AI 改代码，而是先执行 `/opsx:propose`。AI 可以读取仓库、已有 specs、AGENTS.md 和 `.harness/context` 生成第一版变更包；人重点 review 业务边界、上下游影响、验收场景和风险点。

openspec/changes/add-coupon-expire-reminder/
  proposal.md   # 为什么做、做什么、不做什么、影响范围
  design.md     # 技术方案、数据变更、幂等、失败处理、风险
  tasks.md      # 可执行任务清单，每一步都能验证
  specs/coupon-reminder/spec.md
                # Requirement + Scenario，作为验收标准
3

让 AI 按 spec 小步实现，并强制找参照模块

进入 `/opsx:apply` 或编码阶段后，要求 AI 先读 `tasks.md`，再找相似模块，最后一项一项实现。不要让一个 agent 一口气改完整条链路；高风险场景可以拆成开发 agent、测试 agent、review agent。

4

合并前门禁：不靠“看起来对”，靠确定性检查

AI 写完以后，企业级 Harness 必须接入 PR/CI 门禁。单测和编译只是底线，还要覆盖接口契约、数据库变更、安全、架构边界和可观测性。能脚本化的规则，不要靠 prompt 让模型记住。

5

预发验证：部署一个真实运行环境

合并前后要有预发、临时 namespace 或 preview 环境，连接尽量接近生产的中间件和配置，但使用隔离数据。这里验证的是服务真的能跑起来，链路真的能走通。

6

用真实流量验证：回放、影子流量、Dark Launch

录制线上流量，在预发或隔离环境回放；或把线上请求复制一份给新版本，只观察新版本结果，不返回给用户。

7

灰度发布：用指标决定是否继续放量

通过 Feature Flag、Canary 和自动回滚，把上线变成 1% 到 100% 的受控过程。每一步看错误率、延迟、慢 SQL、业务指标和上下游失败。

8

复盘写回：把一次问题变成长期能力

AI 犯错、CI 拦截、预发失败、灰度回滚，都不应该只修代码；还要把问题写回 OpenSpec、Harness 规则、测试、hook 或监控，下一次让系统自动拦住。

企业级实践的核心不是“让 AI 多写代码”，而是把 AI 放进一条可审计、可验证、可回滚、可持续学习的研发流水线里。
05

第四步展开：CI/CD 门禁到底检查什么？

CI 管“能不能合并”，CD 管“能不能发布”。门禁的意思是：失败就不让继续往后走。

CI 是 Continuous Integration，持续集成，主要发生在 PR/MR 阶段，用来证明代码能不能安全合进主干。CD 是 Continuous Delivery / Deployment，持续交付或持续部署，主要发生在合并后，用来证明代码能不能安全发布到测试、预发或生产环境。

1

编译/类型

防止少 import、方法签名错、类型不匹配。

2

测试

单测、集成测试、E2E，覆盖 spec 里的场景。

3

契约

OpenAPI / RPC / DTO / 错误码是否兼容。

4

DB/安全

migration dry-run、SQL explain、secret 和依赖漏洞扫描。

5

发布

构建镜像、部署预发、灰度指标、异常回滚。

# .harness/scripts/verify.sh
#!/usr/bin/env bash
set -euo pipefail

mvn clean compile
mvn test
./.harness/scripts/check-api-contract.sh
./.harness/scripts/check-db-migration.sh
./.harness/scripts/check-architecture.sh
./.harness/scripts/check-secrets.sh

echo "verify passed"
一句话：CI/CD 门禁就是把“人和 AI 容易忘的质量规则”变成机器强制执行的检查。
