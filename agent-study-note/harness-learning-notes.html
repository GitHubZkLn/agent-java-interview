Agent Harness Engineering · Study Note

Harness 不是一张大图能讲完的概念，它是一套让 agent 可靠工作的工程系统。

读这篇笔记时，可以先记住一条主线：模型能力提升以后，瓶颈会从“模型会不会回答”转向“agent 能不能在真实环境里稳定行动”。这就是 Harness Engineering 被单独提出的原因。

我把公开资料归纳成 9 个学习模块：OpenAI 起源、概念演进、Agent loop、Prompt / Context / Harness 边界、Agent Harness 结构、ETCLOVG 七层、运行时实践、验证闭环、Benchmark 与回归。Coding Harness 生产实践已经拆成独立专题页，避免和 Agent Harness 概念线混在一起。

C

00

AI 语境里的 Harness 是怎么来的？

这轮热度的起点，是 OpenAI 在 2026 年围绕 Codex 提出的 Harness Engineering。

这里先把时间线说清楚：harness 这个英文词和软件里的 test harness 并不是今年才有；但 Harness Engineering 作为 AI Coding / Agent 时代的一种工程范式，是 OpenAI 在 2026 年 Codex 实践文章里正式带火的。中文社区后来讨论它时，常把它理解成“AI 编程时代，人不再只写代码，而是设计让 agent 高效工作的环境、脚手架和反馈回路”。

OpenAI 那篇文章的重点不是“发明一个新框架”，而是把工程师角色重新描述了一遍：当 Codex 这类 coding agent 能读代码、改代码、跑测试时，人的价值开始转向设计任务环境、定义成功标准、搭建反馈循环、沉淀内部工具。也就是说，真正要卷的不只是模型本身，而是模型外面的工作系统。

OpenAI 的起点

Codex 能执行工程任务后，重点从“亲手写代码”转向“搭环境、给约束、建反馈”。

中文社区的转述

人类更像导演和系统设计者，Agent 负责执行；质量来自环境和闭环。

Survey 的扩展

论文把这种外部系统系统化，拆成 ETCLOVG 七层 agent harness。

为什么叫 Harness？

因为 harness 的底层隐喻一直是“包住一股能力，让它可连接、可牵引、可约束”。在 AI Coding 里，被包住的是 coding agent；在 Agent 开发里，被包住的是完整的 agent run；在 Evaluation 里，被包住的是一组可重复评测的运行过程。名字变了，核心没变：让能力进入可控工程系统。

不同环境下，Harness 指的不是同一个东西

理解 harness 最容易踩的坑，是把它当成一个固定产品名。它更像一种工程角色：不同环境里，harness 包住的对象不同，提供的控制面也不同。

环境 Harness 包住谁？ 主要提供什么？ 典型问题
Agent 开发 LLM agent 的完整执行过程。 工具、上下文、任务 loop、沙箱、trace、eval、治理。 这个 agent 能否在真实任务中稳定行动？
AI Coding coding agent 与代码仓库。 workspace、git、测试命令、依赖安装、PR 流程、代码审查。 它能否安全改代码、跑测试、交付可合并变更？
所以，“harness 是什么”没有脱离上下文的唯一答案。更稳的回答方式是：先说它包住的对象是什么，再说它给这个对象补了哪些工程能力。放在 Agent 项目里，Harness 不是某个单点组件，而是一层运行基底：它定义 Agent 的活动范围，接入工具和外部系统，组织上下文和任务流程，记录每一步执行轨迹，把结果接入验证和反馈循环，同时用权限、审批、审计去约束高风险动作。换句话说，Harness 的价值不是让 Agent 看起来更智能，而是让 Agent 的执行过程变得可运行、可观察、可复盘、可验证、可迭代、可治理。
01

为什么需要 Harness Engineering？

从“单次回答”走向“长时间执行”，问题类型变了。

早期使用 LLM，最常见的问题是回答不稳定，所以大家首先优化 prompt。OpenAI 今年围绕 Codex 提出 Harness Engineering 后，大家开始意识到：当 agent 能读代码、改代码、跑测试，真正要设计的是它的工作环境和反馈回路。后来 agent 又开始检索资料、调用工具、跨多轮执行，问题就不只是模型输入了，而是整个执行系统是否安全、可观测、可验证、可回滚。

论文《Agent Harness Engineering: A Survey》是在这个基础上进一步系统化：它把“包在模型外面的基础设施层”命名为 agent execution harness，并用 ETCLOVG 七层解释生产环境里的可靠性来自哪里。

阶段 1：模型回答

目标是让模型按格式输出，主要靠 prompt、示例、角色和约束。

阶段 2：多轮协作

目标是让模型看到正确材料，主要靠检索、压缩、记忆和上下文预算。

阶段 3：真实行动

目标是让 agent 安全地做事，靠沙箱、工具协议、trace、eval 和治理。

关键理解

Harness Engineering 的对象不是某一句 prompt，而是 agent 的工作系统。它关心的是：agent 在哪里运行、能用什么工具、看到哪些状态、怎样循环推进、怎么记录过程、如何判断结果对不对、哪些动作必须被限制。

02

Harness Engineering 与 Prompt、Context 的关系

三者不是替代关系，而是工程半径逐层外扩。

如果只记一句话：Prompt 管表达，Context 管信息，Harness 管运行系统。Prompt 和 Context 都可以被看作 Harness 的组成部分，但 Harness 还要覆盖执行环境、工具协议、生命周期、可观测、评测和治理。

维度 Prompt Engineering Context Engineering Harness Engineering
核心问题 这次请求怎么说，模型才照做？ 这一轮应该让模型看见哪些信息？ agent 怎样在真实环境里安全、稳定、可验证地执行？
主要对象 system prompt、few-shot、角色、输出格式、推理步骤。 检索片段、历史摘要、工具返回、长期记忆、上下文压缩。 沙箱、工具协议、任务 loop、trace、成本、评测、权限、审批。
典型动作 让模型“按 JSON 输出、先分析再回答、不要编造”。 把报错日志、API 文档、相关代码、历史 issue 放到上下文。 让 agent 在隔离分支修改代码，跑测试，失败回滚，危险操作审批。
失败表现 答偏、格式错、没有遵守口径。 漏读关键资料、引用过期信息、上下文被噪声挤满。 无限循环、误用工具、改坏文件、不可复现、没有验收标准。
一句话 让模型答得像样。 让模型看得明白。 让 agent 做得可靠。
为什么很多人会混淆？

因为早期 agent 框架常把 prompt、memory、tool calling 放在一个抽象里，导致大家误以为“加工具 + 加记忆”就是 harness。实际进入生产后，真正棘手的是状态恢复、权限边界、trace 诊断、回归评估和上线门禁，这些已经超出 prompt/context 本身。

03

用一个修 Bug 任务来理解三者区别

例子比定义更好记。

假设你让一个 coding agent 处理“接口偶发 500”。同样是这个任务，三种工程会从不同层面介入。

只做 Prompt

你会写：请先分析日志，再定位代码，最后给出修复方案和测试建议。它改善的是回答结构，但并不保证 agent 真能安全改代码。

做 Context

你会把错误日志、trace id、接口文档、相关类、最近提交和相似 issue 放进上下文。它改善的是判断材料。

做 Harness

你会给 agent 隔离 workspace、工具权限、测试命令、变更审计、CI gate 和回滚策略。它改善的是执行可靠性。

Prompt 层会问

“我怎么让模型按排障专家的方式回答？”

Context 层会问

“我应该给它哪些日志、代码、文档和历史信息，避免它猜？”

Harness 层会问

“它可以改哪些文件？测试怎么跑？失败怎么停？过程怎么复盘？上线前谁来验收？”

所以，Prompt 和 Context 可以提升 agent 的判断质量；Harness 决定这个判断能不能变成可控、可验收、可持续迭代的行动。

Agent Harness 到底包住了什么？

把它看成模型外面的任务执行基础设施。

Agent Harness 不是单个库，也不是某个“智能体框架”的同义词。更准确地说，它是一组运行时能力的组合：让 agent 能够接收目标、读取状态、调用工具、产生副作用、记录轨迹、接受评测，并在必要时被人类或策略系统拦住。

用户目标

任务、约束、验收标准、风险等级。

Agent Loop

计划、行动、观察、反思、继续或停止。

Harness Control

沙箱、工具、上下文、trace、eval、权限。

如果没有 Harness

agent 可能能生成一个看似合理的计划，但它改了什么、为什么失败、能不能回滚、是否越权、下一版是否更好，都很难说清楚。

如果有 Harness

每次运行都被约束在明确环境里，工具调用有契约，过程有 trace，输出有验收，风险动作有审批，失败可以归因和复现。

06

ETCLOVG：论文的七层分类

这是读 Survey 最值得带走的框架。

论文把 agent harness 拆成七层：Execution、Tooling、Context、Lifecycle、Observability、Verification、Governance。前四层偏“结构核心”，后三层偏“控制平面”。这个拆法的价值在于：你可以用它检查一个 agent 系统到底缺哪块。

E

执行环境。决定代理的运行位置以及限制其运行的沙箱约束，包括托管沙箱、微型虚拟机、代码专用运行时、计算机使用环境、浏览器沙箱和操作系统级权限模型。

T

工具接口和协议。规定如何描述、发现和调用外部功能，包括协议标准、工具描述和选择、工具增强型训练以及会话管理。

C

上下文和内存管理。控制模型在短期、会话级和持久范围内可以看到的内容，包括长期上下文技术和上下文漂移的缓解措施。

L

生命周期和编排。组织读取和写入状态的控制流，从单代理内部循环到多代理模式，以及完整的从问题到拉取请求的任务管道。

O

可观察性和运维。通过追踪平台、代理专用运维工具、成本跟踪和统一可观测性，捕获追踪、成本、故障和可靠性信号。

V

验证和评估。将任务和跟踪转化为评估、故障归因和回归反馈，包括基准测试、受控执行、多级判断和部署时评估循环。

G

治理与安全。约束模型级、系统级和组织级子层的行为，包括权限模型、生命周期钩子、组件加固、声明式规则和审计基础设施。

怎么用七层做系统检查？

拿一个 agent 项目，从 E 到 G 逐层问：环境隔离了吗？工具契约清楚吗？上下文有来源和过期策略吗？任务能停止和恢复吗？trace 能复盘吗？结果能回归验证吗？高风险动作有审批和审计吗？这些问题比单纯问“用了哪个模型”更能判断工程成熟度。

07

验证闭环：Harness 怎么证明自己有效？

这是 Harness 的一个重要模块，不是整篇笔记的主角。

验证闭环是 Harness Engineering 里不可缺的一块，但它只是整个 harness 的 V 层。它不是只跑公开榜单，而是围绕你的真实任务建立用例、运行记录、评分、baseline 对比和 CI gate。它解决的问题是：这次改了 prompt、工具、上下文或策略之后，agent 是真的更好，还是只是在某几个例子上看起来更顺？

1

Eval Case

准备输入、期望输出、验收标准和风险标签。

2

Agent Run

按用例运行 agent，保留完整执行轨迹。

3

Episode

打包日志、工具调用、输出、耗时、成本。

4

Score

多维评分，归因失败，观察趋势。

5

Gate

达标放行；不达标回归、降级或人工复核。

对比 Benchmark Evaluation Harness 怎么用
任务来源 公开任务集，如 SWE-bench、WebArena、GAIA。 你的真实业务任务、历史事故、真实用户路径。 Benchmark 用来横向了解能力；Eval Harness 用来上线前防回归。
核心问题 我在公共榜单排第几？ 这次改动有没有让我的 agent 变好或变差？ 不要只追榜单，要建立自己的 baseline。
08

口述高分答案：怎么讲 Agent Harness？

适合面试、复盘或讲给同学听。

30 秒版本

Agent Harness 是包在 LLM agent 外面的执行基础设施。它把模型、工具、上下文、任务循环、日志、评测和权限组织起来，让 agent 不只是“能回答”，而是能在真实环境里可靠执行。

1 分钟版本

Prompt Engineering 解决“怎么说”，Context Engineering 解决“给模型看什么”，Harness Engineering 解决“agent 怎么跑”。比如 coding agent 修 bug，prompt 可以规定排障步骤，context 可以提供日志和代码，harness 则要提供隔离 workspace、工具调用协议、测试命令、trace、CI gate 和权限边界。生产里很多失败不是模型不会，而是执行环境、状态恢复、工具契约、验证闭环和治理没做好。

常见误区

误区一：Harness 等于 agent 框架。其实框架偏开发抽象，Harness 更偏生产运行系统。误区二：Context 做好了就够。其实上下文只是 C 层，还缺 E/T/L/O/V/G。误区三：模型越强 Harness 越不重要。模型变强会让部分包装变薄，但只要 agent 有现实副作用，就需要边界、记录和验收。

Harness 高频面试题

围绕概念、问题、区别和 Agent Harness 设计四个核心问题准备。

1. 什么是 Harness Engineering？

答题要点：Harness Engineering 是 AI Coding / Agent 时代的一种工程方法：不只优化模型提示词，而是设计包在 agent 外面的工作系统，包括运行环境、工具接口、上下文供给、任务流程、反馈闭环、验证和权限控制。它的目标是让 agent 的能力可连接、可观察、可验证、可治理。

2. Harness Engineering 解决什么问题？

答题要点：它解决的是 agent 从“能回答”到“能可靠执行”的工程问题。比如：在哪里运行、能用哪些工具、怎么控制上下文、如何避免无限循环、怎么记录过程、怎么判断结果正确、失败如何回滚、高风险操作如何审批。生产里很多问题不是模型不会，而是执行环境、反馈闭环和治理没设计好。

3. Harness Engineering 和 Prompt Engineering、Context Engineering 的区别？

答题要点：Prompt Engineering 解决“怎么说”，通过指令、示例、格式约束提升单次回答质量；Context Engineering 解决“给模型看什么”，通过检索、记忆、压缩和工具结果管理提升判断质量；Harness Engineering 解决“agent 怎么跑”，把模型、工具、状态、流程、观测、验证和权限组织成一个可运行系统。三者不是替代关系，Harness 包含并协调 Prompt 与 Context。

4. 设计 Agent Harness 时应该考虑哪些方面？

答题要点：可以按 ETCLOVG 七层回答：E 看执行环境和权限边界；T 看工具接口和协议；C 看上下文和记忆管理；L 看任务生命周期和编排；O 看 trace、日志、成本和可观测；V 看测试、评估和回归；G 看治理、安全、审批和审计。面试时最好结合一个场景，比如 coding agent：隔离 workspace、工具白名单、上下文检索、最大轮数、trace、CI gate、敏感文件保护。
