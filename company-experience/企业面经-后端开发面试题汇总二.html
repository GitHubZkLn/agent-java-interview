Redis 7 题
Q1
Redis 支持哪些数据结构？各自的底层实现是什么？
▼

Redis 支持的 5 种基本数据结构 + 3 种特殊数据结构：

  • String（字符串）：底层实现为 SDS（Simple Dynamic String），即动态字符串。SDS 记录了字符串长度，支持 O(1) 的 strlen 操作；空间预分配 + 惰性释放，避免频繁内存分配；可存储字符串、整数、浮点数，也可用于缓存序列化对象。
  • List（列表）：底层实现为 QuickList（Redis 3.2 之后），即多个 ZipList 组成的双向链表。ZipList 是一片连续内存（压缩列表），节省内存；当列表数据量小、元素短时使用 ZipList，数据量大了则转为 QuickList。
  • Hash（哈希）：底层实现为 ZipList（数据量小）或 HT（HashTable）。当 Hash 字段数量少、值短时用 ZipList；超过阈值后转为 HashTable（渐进式 rehash）。
  • Set（集合）：底层实现为 IntSet（整数集合）或 HT（HashTable）。当集合内全为整数且数量少时用 IntSet（紧凑数组）；否则用 HashTable（value 恒为 NULL）。
  • ZSet（有序集合）：底层实现为 ZipList 或 SkipList + HT。当元素少时用 ZipList（元素和分值交错存储）；超过阈值后转为 SkipList（跳表）配合 HT 实现。
  • Bitmap（位图）：基于 String 类型实现，用 bit 位存储数据，适合签到、活跃用户统计等场景。
  • HyperLogLog：基于概率算法，用于海量数据下精确度要求不高的基数统计（UV 统计），内存固定 ≈ 12KB。
  • Geospatial（地理空间）：底层使用 ZSet + GeoHash 编码，存储地理位置坐标，支持距离查询、附近的人等功能。
  • Stream：Redis 5.0 引入的持久化消息队列，底层使用 RadixTree（基数树）实现，支持消费者组。
Q2
Redis 的过期策略和淘汰策略是怎样的？
▼

过期策略（删除过期 key 的时机）：

  • 惰性删除（Lazy Expire）：访问 key 时才检查是否过期，过期则删除。优点是对 CPU 友好（只删除必须删除的），缺点是过期 key 堆积可能造成内存压力。
  • 定期删除（Active Expire）：Redis 每隔一段时间随机抽取一批设置了过期时间的 key 检查删除（通过 ACTIVE_EXPIRE_CYCLE_SLOW_TIME_PERC 配置扫描时间比例）。折中了 CPU 和内存之间的平衡。
  • 惰性删除 + 定期删除 组合使用，是 Redis 的默认过期策略。

淘汰策略（内存达到 maxmemory 时的处理方式）：

  • noeviction：不淘汰，返回错误（默认策略）
  • volatile-lru / allkeys-lru：LRU 算法淘汰（最近最少使用）。volatile-lru 只淘汰设置了过期时间的 key；allkeys-lru 淘汰所有 key，最常用
  • volatile-lfu / allkeys-lfu：LFU 算法淘汰（最不经常使用）。按访问频率淘汰
  • volatile-random / allkeys-random：随机淘汰。volatile-random 只淘汰设置了过期时间的；allkeys-random 淘汰所有
  • volatile-ttl：淘汰 TTL 最小的 key（即最快过期的）

选型建议：一般生产环境使用 allkeys-lru（内存不够时自动淘汰最久未使用的 key）；如果业务需要保留热点数据但允许被淘汰，使用 volatile-lru。

Q3
Redis 如何实现分布式锁？有什么缺陷？如何优化？
▼

基础实现（SET NX EX）：

使用 SET lock_key unique_value NX EX expire_time 原子性地获取锁；释放锁时用 Lua 脚本验证 value 是否一致（防止误删他人的锁）：

if redis.call("get", KEYS[1]) == ARGV[1] then return redis.call("del", KEYS[1]) else return 0 end

基础方案的缺陷：

  • 单机单点故障：Redis 主从架构下，主从异步复制可能导致锁丢失（主节点加锁成功但从节点未同步，主节点故障后从节点升主，锁丢失）
  • 锁过期时间难以确定：业务执行时间可能超过锁 TTL，导致锁被他人获取
  • 无法等待锁：没有公平的等待队列机制

优化方案：

  • Redlock（红锁）：向 N 个独立 Redis 实例（奇数个，如 5 个）同时用相同 key + value 获取锁，超过 N/2+1 个成功则视为获取成功。提高了可靠性，但实现复杂且性能开销大。
  • 看门狗（Watchdog）：如 Redisson 方案，持有锁的线程启动一个后台定时任务，每隔 TTL/3 时间自动续期（延长过期时间），避免业务未执行完锁就过期。
  • Zookeeper 实现分布式锁：利用 ZK 的临时有序节点和 Watch 机制，有公平等待队列，锁可靠性更高，但性能低于 Redis。
  • 数据库唯一索引：利用数据库唯一约束实现分布式锁，简单但性能差、无法处理锁重入。
Q4
Redis 持久化机制有哪些？RDB 和 AOF 的区别是什么？
▼

RDB（Redis Database）—— 快照持久化：

通过 BGSAVE 命令（子进程执行）或 SAVE 命令（阻塞主进程），将 Redis 在某个时间点的全量数据写入磁盘的 dump.rdb 文件。触发方式：定时任务（默认 bgrewriteaof 触发）、手动 SAVE/BGSAVE、shutdown 时。

AOF（Append Only File）—— 操作日志持久化：

Redis 执行的每一个写操作命令都追加到 appendonly.aof 文件末尾，重启时通过重放这些命令恢复数据。默认关闭，需设置 appendonly yes 开启。

RDB vs AOF 对比：

  • 文件大小：RDB 是压缩的二进制文件（紧凑），体积小，恢复快；AOF 记录的是操作命令文本，文件大，恢复慢
  • 数据完整性：RDB 丢失数据多（取决于上次备份到故障之间的数据）；AOF 可以配置每秒同步（everysec）或每条命令同步（always），丢失数据少
  • 性能影响：RDB fork 子进程，不影响主进程性能，但 fork 时阻塞；AOF 每秒 fsync 一次对性能影响小
  • 优先级：同时开启时，Redis 重启优先使用 AOF 恢复（数据更完整）

AOF 重写（Rewrite）：AOF 文件过大会触发 bgrewriteaof，通过后台进程压缩命令（合并多条写命令为一条），减少文件体积。

生产环境推荐配置：RDB + AOF 都开启，AOF 策略设为 everysec（每秒落盘一次，兼顾性能和数据安全）。

Q5
Redis 主从复制原理是什么？
▼

Redis 主从复制用于读写分离和数据冗余，异步进行。流程分为全量同步和增量同步两个阶段：

全量同步（第一次同步 / 主从不一致时）：

  • 从库发送 PSYNC ? -1（或 PSYNC <runid> <offset>）给主库
  • 主库收到后执行 BGSAVE 生成 RDB 快照，同时将后续写命令缓存到 replication buffer
  • RDB 写完后，主库将 RDB 文件发送给从库（replication buffer 和 replication backlog 配合实现增量衔接）
  • 从库接收 RDB 并加载到内存，然后等待主库将缓存期间的增量命令发送过来执行

增量同步（命令传播阶段）：

主库每执行一个写命令，就异步地将命令发送给所有从库。从库每秒上报自己的 repl_offset（已接收命令的偏移量），主库与从库的差距记录在 replication backlog 环形缓冲区中。如果从库网络断开后重新连接，且差距在 backlog 范围内，则只同步增量命令（PSYNC <runid> <offset>）。

重要配置：replica-serve-stale-data（从库过期或主从不一致时的行为）、replica-read-only（从库只读）、repl-backlog-size（backlog 缓冲区大小，需设置足够大避免增量同步失败）。

Q6
Redis 缓存穿透、缓存击穿、缓存雪崩是什么？如何解决？
▼

1. 缓存穿透（Cache Penetration）：

大量请求查询数据库中根本不存在的数据（恶意或非恶意），缓存中没有，数据库中也没有，请求全部打到数据库。

解决方案：

  • 布隆过滤器（最推荐）：将所有合法数据 key 存入布隆过滤器，请求进来先过布隆过滤器，在则查缓存；判断不存在则直接返回
  • 空对象缓存：将 NULL 结果也缓存起来（如 SET key null 5min），短时间内相同非法 key 直接返回缓存结果
  • 参数校验：在接口层对请求参数做基础校验（如 id > 0）

2. 缓存击穿（Cache Breakdown）：

某个热点 key 过期瞬间，大量并发请求同时涌入查询该 key，全部穿透到数据库。

解决方案：

  • 互斥锁 / 分布式锁：加锁只允许一个线程去查数据库并回填缓存，其他线程等待（Redisson RLock）
  • 热点数据永不过期：不设置 TTL，而是通过后台异步更新；或者逻辑过期（存两个字段：过期时间 + 值，访问时判断逻辑过期则异步重建）

3. 缓存雪崩（Cache Avalanche）：

大量 key 同时过期，或 Redis 突然宕机，导致大量请求同时打到数据库。

解决方案：

  • 过期时间分散：在 TTL 基础上加上随机偏移量（TTL + random(0, TTL/2)），避免同时过期
  • 多级缓存：本地缓存（Caffeine/Guava Cache）+ Redis + 数据库，多一层保护
  • Redis 高可用：主从 + Sentinel / Cluster，避免全量故障
  • 服务熔断/限流：缓存服务不可用时，通过 Hystrix / Sentinel 等限流，避免数据库被打垮
Q7
Redis 和 Memcache 的区别是什么？
▼
  • 数据结构：Redis 支持 String、List、Hash、Set、ZSet、Bitmap、HyperLogLog、Stream 等丰富数据结构；Memcache 只支持简单的 Key-Value（Value 最大 1MB，不支持数据结构）。
  • 持久化：Redis 支持 RDB + AOF 两种持久化方式，可将数据持久化到磁盘；Memcache 不支持持久化，重启后数据丢失。
  • 主从复制：Redis 支持主从复制、哨兵、集群；Memcache 不支持复制。
  • 内存管理：Redis 使用自己的内存分配器（jemalloc），支持内存碎片管理；Memcache 使用 slab 机制（固定大小的内存块分配）。
  • 单线程 vs 多线程：Redis 6.0 之前是单线程（6.0 后引入了 IO 多线程），单线程避免了锁开销，命令执行是原子的；Memcache 是多线程模型（Memcached 1.6+，使用 libevent）。
  • 应用场景：Redis 适合需要丰富数据结构、持久化、主从复制的场景（缓存、分布式锁、消息队列、计数器）；Memcache 适合简单的 KV 缓存场景，对可靠性要求不高的情况。

消息队列 7 题
Q1
消息队列的应用场景有哪些？有什么优缺点？
▼

消息队列的核心价值：解耦、异步、削峰。

典型应用场景：

  • 异步处理：将非核心流程异步化，提升系统响应速度（如用户注册后发送邮件/短信通知、订单完成后触发积分/库存操作）。原本串行的"注册+发邮件+发短信"变为注册完成后直接返回，MQ 异步通知各下游。
  • 流量削峰：在大促/秒杀场景下，将突增流量先写入 MQ，下游服务按自身能力从 MQ 消费，避免压垮数据库。
  • 系统解耦：微服务间通过消息总线通信，上下游无需直接调用，新增/修改消费者不影响生产者（如支付系统通知库存、物流、积分等多个下游）。
  • 日志处理：将日志采集到 Kafka，集中处理（ETL、实时分析、归档）。
  • 分布式事务一致性：通过 MQ + 事务消息（RocketMQ）实现最终一致性。

优点：解耦、削峰、异步；提升系统吞吐量和稳定性；支持一对多广播。

缺点：

  • 系统可用性降低（MQ 挂了影响全局）
  • 系统复杂度提升（消息丢失、重复消费、顺序性问题需要处理）
  • 数据一致性问题（分布式环境下无法保证强一致性，只能最终一致）
Q2
Kafka、RabbitMQ、RocketMQ 有什么区别？如何选型？
▼

Kafka（LinkedIn 开源，流处理场景首选）：

  • 架构：分布式、分区（Partition）+ 多副本（Replication）。每个 Partition 顺序写磁盘，顺序读写性能极高。
  • 消息模型：基于拉取（Pull）的消费模式，消费者自己控制消费进度（offset）。
  • 吞吐量：最高（百万级 QPS），适合大数据、日志采集、流处理。
  • 消息持久化：天然持久化，消息存储在磁盘，顺序写入，IO 效率高。
  • 消息顺序性：单 Partition 内有序；不保证跨 Partition 有序。
  • 不支持事务消息。

RabbitMQ（ Erlang 开发，功能最丰富）：

  • 架构：Exchange（交换机）+ Binding + Queue。消息先到 Exchange，根据路由键转发到不同 Queue。支持 4 种 Exchange 类型：Direct、Fanout、Topic、Headers。
  • 消息模型：推（Push）模式，Broker 主动推送给消费者。
  • 吞吐量：中等（万级 QPS）。
  • 消息可靠性：支持消息确认（ACK）、死信队列（DLX）、延迟队列。
  • 不支持消息回溯（重新消费历史消息）。

RocketMQ（阿里开源，电商场景）：

  • 架构：NameServer 集群管理路由信息，Broker 分为 Master/Slave。
  • 核心优势：支持事务消息（半消息 + 回查机制），实现分布式事务；支持消息回溯（按时间戳或 offset 重置消费位点）。
  • 吞吐量：高（十万级 QPS）。
  • 顺序消息：支持队列维度的严格顺序。

选型建议：日志采集 / 大数据流处理选 Kafka；业务系统需要丰富路由规则、延迟队列、死信队列选 RabbitMQ；需要事务消息、订单场景选 RocketMQ。

Q3
如何保证消息的可靠传输（不丢失）？
▼

消息从生产者到消费者，中间经历三个环节，每个环节都可能丢消息，需要逐一保障：

1. 生产者到 Broker（发送阶段）：

  • 使用同步发送 + 回调确认：发送后等待 Broker 确认（ACK），失败则重试（Kafka 的 retries 配置，RabbitMQ 的 publisher confirm）。
  • Kafka 设置 acks=all（所有 ISR 副本确认）+ min.insync.replicas=2，确保消息写入多个副本。
  • RabbitMQ 使用事务模式（channel.txSelect()）或 publisher confirm 机制。

2. Broker 存储阶段：

  • Broker 开启同步刷盘（flush to disk）配置，不要使用异步刷盘（性能优先时可接受少量丢失）。
  • Kafka 设置 replication.factor >= 3（多副本）、unclean.leader.election.enable=false（不允许落后太多的副本当选 Leader）。
  • RabbitMQ 开启持久化（Queue durable=true + Message persistent=true）。

3. Broker 到消费者（消费阶段）：

  • 使用手动 ACK（手动确认），业务处理完成后才发送 ACK，不要使用自动 ACK（Auto ACK）。
  • 如果处理失败，不 ACK，消息会重新投递给其他消费者（或重试队列）。
  • 消费端做好幂等性处理（重复消费的场景下保证结果正确）。
Q4
如何保证消息的顺序性？
▼

消息顺序性分为两种：全局顺序和局部顺序。实际场景中通常只需要保证局部顺序（如同一订单的消息按先后顺序处理）。

全局顺序：所有消息严格按发送顺序消费。实现方式：Topic 只建 1 个 Queue，生产者和消费者都单线程。单线程瓶颈大，实际很少使用。

局部顺序（通常场景）：同一类别的消息（如同一订单、同一用户）按顺序处理，不同类别之间无顺序要求。

实现方案：

  • Kafka：将需要保序的消息发到同一个 Partition（按消息 key 做哈希取模路由）。Kafka 保证同一 Partition 内消息有序。由于 Partition 可并行消费，整体吞吐量不受影响。
  • RabbitMQ：将需要保序的消息路由到同一个 Queue（用同一个 routing key）。消费者端使用单线程消费该 Queue（或使用分布式锁 + 多线程处理，同一 key 的消息串行处理）。
  • 消费者端：如果需要多消费者并行处理，可在消费者端按消息 key 哈希后分发到不同的内存队列，每个内存队列单线程处理。

注意事项：

  • 不要将大量消息路由到同一个 Partition（热点问题）
  • 消费者崩溃重启后，需要从上次消费的 offset 继续（确保不丢消息也不重复消费）
Q5
如何处理消息的重复消费？
▼

消息重复消费的本质是消息可靠性要求高时（ACK 前重试），消费者需要做好幂等性处理。以下是常用方案：

1. 数据库唯一索引 / 主键约束（最常用）：

将消息 ID 作为表的主键或唯一索引。由于重复消息的 ID 相同，插入时会报主键冲突。处理逻辑：INSERT 时使用 ON DUPLICATE KEY UPDATE 或先查再决定是否插入。

2. 分布式锁：

消费前用消息 ID 申请分布式锁（Redis SETNX），抢到锁的线程执行消费逻辑，其他线程跳过。锁自动过期避免死锁。

3. 去重表（Lookup Table）：

维护一张去重表（如 Redis SET，key = 消息ID），消费前先查是否已处理过。适用于实时性要求高的场景。

4. 业务状态机：

如果消息触发的操作是状态变更（如订单状态：待支付 → 已支付 → 已完成），通过状态机判断：只有当前状态允许转换时才执行，否则跳过。

5. 消息去重窗口：

使用 Redis 记录最近 N 分钟内已消费的消息 ID（设置过期时间 TTL），每次消费前先检查。

注意事项：幂等性应由消费端保障，而非依赖 MQ 本身去重（MQ 不保证 exactly-once）。不同业务场景选择最合适的幂等方案即可。

Q6
Kafka 的架构是怎样的？为什么吞吐量高？
▼

Kafka 核心架构：

  • Topic（主题）：消息的逻辑分类，每个 Topic 可分为多个 Partition（分区）
  • Partition（分区）：每个 Partition 存储在不同的 Broker 上，实现数据分布式存储；每个 Partition 有多个副本（ISR：In-Sync Replicas）
  • Producer（生产者）：按 key 哈希路由到指定 Partition；可配置 acks 级别
  • Consumer（消费者）：属于 Consumer Group，同一 Group 内不同 Consumer 消费不同 Partition；不同 Group 可重复消费同一消息
  • ZooKeeper / KRaft：早期版本用 ZK 管理 Broker 注册、分区 Leader 选举；Kafka 3.0 后支持 KRaft（无需 ZK）

Kafka 高吞吐量的原因：

  • 顺序写磁盘：Kafka 将消息追加到文件末尾（Append-only log），磁盘顺序写入速度极快（接近内存速度），远高于随机写入。
  • 零拷贝（Zero-Copy）：使用 Linux 的 sendfile() 系统调用，数据从磁盘到网络的传输绕过用户态，直接由内核态完成，减少了 2 次 CPU 拷贝。
  • 批量处理：多条消息打包成批次（Batch）后发送和存储，减少网络请求次数和磁盘 I/O。
  • 数据压缩：支持批量压缩（Batch GZIP/Snappy/LZ4/ZSTD），压缩后传输和存储。
  • 页缓存（Page Cache）：利用操作系统的 Page Cache 缓存热点数据，写入时先写 Page Cache 由 OS 异步刷盘，读取时优先读缓存。
  • 多副本 + 分区并行：横向扩展，多个 Partition 分散到不同机器并行读写。
Q7
RabbitMQ 的 Exchange 类型有哪些？各自的区别是什么？
▼

RabbitMQ 的消息路由模型为：Producer → Exchange → Binding → Queue → Consumer。Exchange 根据路由键（Routing Key）和 Binding 的规则，将消息路由到一个或多个 Queue。

四种 Exchange 类型：

  • Direct（直连交换机）：完全匹配路由键。Binding 时指定完整的 routing key，消息的 routing key 必须与 binding key 完全一致才路由到对应 Queue。适用于一对一精准路由（如不同优先级任务分发）。
  • Fanout（广播交换机）：忽略路由键，将消息广播到所有绑定的 Queue。相当于子网广播，所有消费者都会收到消息副本。适用于群发通知、多个消费者独立处理同一任务。
  • Topic（主题交换机）：按通配符匹配路由键。* 匹配一个词，# 匹配零个或多个词。routing key 格式为 word1.word2.word3。适用于灵活的消息分类订阅（如 stock.*.usd 可匹配 stock.ba.usd 和 stock.goog.usd）。
  • Headers（头交换机）：根据消息头部的多个属性（键值对）进行匹配，而非 routing key。用得较少，适合需要多维度匹配的场景。

微服务与架构 7 题
Q1
SOA 和微服务架构有什么区别？
▼

SOA（Service-Oriented Architecture，面向服务架构）：

  • 强调整体集成，以企业服务总线（ESB） 为中心，所有服务通过 ESB 通信（HTTP、SOAP、JMS 等协议）
  • 服务粒度较粗，通常是企业级的大业务模块（"用户服务"、"订单服务"）
  • ESB 承担协议转换、消息路由、服务注册发现等，功能重，容易成为单点瓶颈
  • 部署周期长，通常整体部署，灵活度低

微服务架构：

  • 去中心化，每个服务独立进程，独立部署，数据自主管理（每个服务有独立的数据库）
  • 服务粒度细（单一职责），一个微服务只做一件事（如"用户注册"、"发送验证码"、"积分查询"）
  • 服务间通过轻量协议通信（HTTP REST、gRPC、GraphQL），无需 ESB
  • 独立技术栈，不同服务可根据需求选择不同语言和框架
  • 快速迭代，独立发布上线，CI/CD 友好

核心区别：

  • 集成方式：SOA 用 ESB（中心化）；微服务用轻量通信（去中心化）
  • 服务粒度：SOA 粗，微服务细
  • 数据库：SOA 倾向共享数据库；微服务各服务独立数据库
  • 部署：SOA 整体部署；微服务独立部署
  • 适用场景：SOA 适合企业级异构系统集成；微服务适合互联网敏捷开发
Q2
服务注册与发现是如何实现的？Eureka 和 Nacos 有什么区别？
▼

服务注册与发现的核心机制：

服务启动时向注册中心注册自己的 IP、Port、服务名等信息；消费者从注册中心发现可用服务实例列表；心跳机制维持服务实例的存活状态（心跳超时则从注册中心剔除）。

Eureka（Netflix，开源已停止维护）：

  • 架构：每个 Eureka Server 既作为注册中心又作为客户端（去中心化），通过互相注册实现高可用。
  • 保护模式：Eureka Server 在网络分区故障时开启保护模式（self-preservation mode），不会立即剔除心跳过期的服务，避免误删。
  • CAP 取舍：优先A（可用性）和 P（分区容错），允许数据短暂不一致（Eureka 2.0 已停止开发）。
  • 不支持配置管理，不支持命名空间分组，不支持动态配置。

Nacos（阿里巴巴，开源活跃）：

  • 双重身份：既是注册中心又是配置中心（Config Server），一站式解决方案。
  • 注册中心一致性协议：支持两种模式 —— CP（Raft 协议）（使用 Leader 复制，强一致）和 AP（Distro 协议）（无 Leader，最终一致）。可根据场景切换。
  • 命名空间 / 分组：支持 Namespace（环境隔离）、Group（服务分组）、Cluster（集群分组）三层结构。
  • 动态配置：配置变更实时推送到客户端（长轮询 + UDP），支持配置的热更新（不重启服务生效）。
  • 健康检查：支持心跳上报、TCP、HTTP、MySQL 等多种检查方式。

选型建议：新项目推荐 Nacos；Spring Cloud Alibaba 生态首选 Nacos。

Q3
分布式事务的解决方案有哪些？
▼

分布式事务的核心问题：一个业务操作涉及多个服务/数据库，如何保证要么全部成功，要么全部失败/回滚。

1. 两阶段提交（2PC，Two-Phase Commit）：

  • 准备阶段：协调者向所有参与者发送 Prepare，所有参与者执行事务（锁定资源）但不提交，返回成功/失败
  • 提交阶段：协调者收到所有成功回应后，发送 Commit 提交事务；任何一个失败则发送 Rollback 回滚
  • 缺点：协调者是单点；如果部分参与者已提交而协调者挂了，会导致数据不一致；锁定资源时间长，性能差

2. 三阶段提交（3PC，Three-Phase Commit）：

引入 PreCommit 阶段和超时机制，在一定程度上改善了 2PC 的阻塞问题，但仍然无法彻底解决数据不一致问题。

3. TCC（Try-Confirm-Cancel，补偿型）：

  • Try：预留资源（如冻结库存、预扣金额）
  • Confirm：确认执行（真正扣库存、转账）
  • Cancel：取消回滚（释放冻结的库存/金额）
  • 需要业务代码实现 Try/Confirm/Cancel，对业务侵入大；但不锁定资源，并发度高

4. 本地消息表（事务消息）：

将消息和业务操作放在同一数据库事务中，通过本地消息表 + 定时任务轮询（或 RocketMQ 事务消息）保证最终一致性。实现简单，适合大多数业务场景。

5. Saga 模式：

将长事务拆分为多个本地短事务，每个事务有对应的补偿操作（正向/逆向）。不支持回滚（只做补偿）。适合长流程业务（旅行预订、订单处理）。

选型建议：强一致性用 2PC/3PC；大多数业务场景用本地消息表或RocketMQ 事务消息；高并发下单场景用 TCC。

Q4
分布式 ID 的生成方案有哪些？
▼

1. UUID / GUID：

  • 128 位，本地生成，无中心依赖
  • 缺点：无序（导致 B+Tree 页分裂）、字符串存储（36 字符 / 128 bit）

2. 雪花算法（Snowflake）—— 最常用：

  • 64 位 Long：1 位符号 + 41 位时间戳 + 10 位机器 ID + 12 位序列号
  • 趋势递增、高性能、本地生成；依赖机器时钟（时钟回拨可能重复）
  • 变种：美团 Leaf（双 Buffer + 趋势递增优化）、百度 UidGenerator（减少时间回拨风险）

3. 数据库自增：

  • 单表或单库可用 AUTO_INCREMENT，ID 连续性好
  • 分库分表后需要单独的自增序列表或数据库集群（如 MySQL Group Replication）

4. Redis INCR：

  • 利用 INCR 原子性生成 ID，可按业务前缀分区
  • ID 连续性好，但依赖 Redis（需考虑持久化和高可用）

5. 号段模式（Segment）：

  • 批量从数据库/Redis 申请号段（批量获取 1000 个），本地内存分配，用完再申请
  • 减少数据库交互次数，性能好，且 ID 不会断号
  • 代表：美团 Leaf、滴滴（TinyID）

选型：一般业务场景选雪花算法（变种）或号段模式；需要严格顺序号选数据库自增；简单场景可接受字符串用UUID。

Q6
接口的幂等性如何保证？
▼

幂等性：多次执行与一次执行结果相同。常见场景：支付扣款、订单创建、库存扣减。

1. 基于 Token + 业务去重：

客户端发起请求前先从服务端获取唯一 Token（UUID），请求时携带 Token；服务端以 Token 作为 Redis Key（设置 TTL）判断是否已处理。简单有效，但需要额外一次请求获取 Token。

2. 基于数据库唯一索引 / 主键约束：

将幂等 Key（如订单号、用户ID+业务类型+日期）作为唯一索引或主键。重复插入时数据库报主键冲突，捕获异常返回成功即可。适合创建类操作（用户注册、订单创建）。

3. 状态机幂等：

对于状态流转类操作（如订单支付），通过状态机判断：只有当前状态允许转换时才执行。例如：订单状态为"待支付"才允许执行"支付"操作，已支付则跳过。

4. 悲观锁（SELECT FOR UPDATE）：

数据库行锁保证并发安全，SELECT * FROM orders WHERE id = ? FOR UPDATE，加锁后检查状态再执行。

5. 乐观锁（版本号 / CAS）：

UPDATE 时带 version 条件：UPDATE orders SET status='已支付', version=version+1 WHERE id=? AND version=?，影响行数为 0 说明已被他人修改，触发重试。

6. 分布式锁：

用业务 ID 做 Redis 分布式锁（SETNX），同一 ID 并发时只有一个获取锁成功并执行，其他等待或跳过。

设计建议：GET 操作天然幂等；PUT/DELETE 需根据语义设计幂等；POST（创建）通常用唯一索引；状态变更用状态机。

Spring Boot 自动装配的原理是什么？
▼

Spring Boot 的自动装配（Auto Configuration）是其"约定大于配置"理念的核心，让开发者无需手动配置大量 Bean，框架自动根据 classpath 中的依赖和配置文件完成 Bean 的注册。

核心原理：

  • @SpringBootApplication：由 @SpringBootConfiguration（= @Configuration）+ @EnableAutoConfiguration + @ComponentScan 三个注解组成。
  • @EnableAutoConfiguration：通过 @Import(AutoConfigurationImportSelector.class) 导入自动配置选择器。
  • AutoConfigurationImportSelector：重写了 selectImports() 方法。它扫描所有 META-INF/spring.factories（Spring Boot 2.7 之前）或 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports（Spring Boot 2.7+）文件，读取其中声明的自动配置类（FulFillAutoConfiguration）。
  • 条件装配（@Conditional）：每个自动配置类通过 @ConditionalOnClass（classpath 存在某类才生效）、@ConditionalOnProperty（配置项存在且为 true 才生效）、@ConditionalOnBean（容器中不存在某 Bean 才生效）等条件注解，实现"智能"装配——只有当特定依赖存在时才注册对应 Bean。

举例：当 classpath 中有 spring-boot-starter-data-redis 时，RedisAutoConfiguration 上的 @ConditionalOnClass(RedisTemplate.class) 条件满足，框架自动注册 RedisTemplate 等 Bean，无需手动配置。

排除自动配置：@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class}) 或在配置文件中 spring.autoconfigure.exclude。

并发编程 5 题
Q1
ThreadLocal 的原理是什么？存在哪些问题？
▼

原理：

每个 Thread 对象内部有一个 ThreadLocalMap（本质是 Entry[] 数组），key 是 ThreadLocal 对象（弱引用），value 是存入的值。调用 set(T value) 时，实际上是获取当前线程的 ThreadLocalMap，以当前 ThreadLocal 对象为 key 存入 Entry。不同线程访问同一个 ThreadLocal，get 的是各自线程独立的数据副本。

内存泄漏问题（重点）：

ThreadLocalMap 的 Entry 继承自 WeakReference<ThreadLocal<?>>，key（ThreadLocal 对象）是弱引用。当 ThreadLocal 对象没有强引用只剩弱引用时，下次 GC 就会被回收，key 变为 null。但 value 是强引用，如果不为 null，就无法被回收，导致value 内存泄漏。

ThreadLocal 的设计者在 set/get/remove 方法中都做了探测式清理（expungeStaleEntry）：访问时发现 stale key（key == null）就清理掉对应的 value。但更安全的做法是：使用完毕后主动调用 remove()，而不是依赖自动清理。

应用场景：

  • 数据库连接（Connection）在线程间隔离
  • 用户登录 Session 传递（Web 项目中，在拦截器设置 ThreadLocal，获取用户信息时直接取 ThreadLocal）
  • 线程上下文传递（如 TraceId 日志追踪）
  • Spring 事务管理器（TransactionSynchronizationManager）

ThreadLocal vs 继承（InheritableThreadLocal）：普通 ThreadLocal 只在当前线程有效；InheritableThreadLocal 允许子线程继承父线程的值，但需要注意子线程生命周期短时同样有内存泄漏风险。

Q2
synchronized 和 ReentrantLock 的区别是什么？
▼

共同点：都是可重入锁（同一线程可重复获取），保证互斥访问和可见性。

核心区别：

  • 底层实现：synchronized 是 JVM 内置关键字，由 JVM 底层实现（对象头 Mark Word + Monitor）；ReentrantLock 是 java.util.concurrent.locks 包下的类，基于 AQS（AbstractQueuedSynchronizer）实现。
  • 等待方式：synchronized 无法中断等待锁的线程，锁获取一直阻塞；ReentrantLock 可以通过 lockInterruptibly() 中断等待。
  • 公平性：synchronized 是非公平锁（插队式）；ReentrantLock 可以通过构造函数指定公平/非公平（new ReentrantLock(true) 为公平锁）。
  • 锁粒度：synchronized 锁粒度粗（整个方法或代码块）；ReentrantLock 可通过 newCondition() 创建多个条件变量（Condition），更精细地控制线程等待/通知。
  • 性能优化：JDK 1.6 之后对 synchronized 做了大量优化（偏向锁、轻量级锁、自旋锁、锁消除、锁粗化），两者性能差距已不大。
  • API 丰富度：ReentrantLock 提供 tryLock()（尝试获取，获取不到立即返回）、tryLock(timeout)（超时等待）、getHoldCount()（当前线程持有锁次数）等 API。

选型建议：简单互斥用 synchronized（更简洁，JVM 原生支持）；需要公平锁、可中断、超时等待、多条件变量时用 ReentrantLock。

Q3
线程池的核心参数有哪些？拒绝策略是什么？
▼

ThreadPoolExecutor 构造方法核心参数：

  • corePoolSize：核心线程数，即使空闲也不会被回收的线程数量
  • maximumPoolSize：最大线程数，核心线程 + 临时线程的最大数量
  • keepAliveTime / unit：非核心线程的空闲存活时间
  • workQueue：任务队列，用于存放等待执行的任务（如 LinkedBlockingQueue、ArrayBlockingQueue、SynchronousQueue）
  • threadFactory：线程工厂，用于创建新线程（可自定义线程名称、优先级、是否守护线程）
  • handler：拒绝策略，当线程池饱和时的处理策略

线程池工作流程：

任务进来 → 核心线程数未满则创建核心线程执行 → 核心线程满则加入任务队列 → 队列满则创建非核心线程（直到 maxPoolSize） → 线程池饱和则执行拒绝策略

4 种拒绝策略：

  • AbortPolicy（默认）：抛出 RejectedExecutionException，拒绝执行并抛异常
  • CallerRunsPolicy：由调用者线程执行任务（主线程），会降低提交速度，不会丢失任务
  • DiscardPolicy：直接丢弃任务，不抛异常
  • DiscardOldestPolicy：丢弃队列中最旧（最老的）的任务，然后尝试重新提交新任务

常见线程池：Executors.newFixedThreadPool()（固定大小，LinkedBlockingQueue）；newCachedThreadPool()（按需创建，SynchronousQueue）；newSingleThreadExecutor()（单线程）；newScheduledThreadPool()（定时任务）。生产环境建议用 ThreadPoolExecutor 直接创建，避免使用 Executors 快捷方法导致队列无限堆积。

Q4
什么是守护线程？Java 中的线程类型有哪些？
▼

守护线程（Daemon Thread）：

守护线程是为用户线程（User Thread）提供后台服务的线程。JVM 的启动线程是非守护线程（User Thread）。当所有非守护线程都结束时，JVM 会立即终止所有守护线程并退出程序。

设置方式：thread.setDaemon(true) 必须在 start() 之前调用。

典型守护线程：垃圾回收线程（GC）、JVM 内部调度线程、Finalizer 线程。线程池中的线程默认是用户线程。

注意：守护线程中 finally 块可能不执行（JVM 突然终止）；不要用守护线程处理数据库连接、文件读写等需要清理资源的工作。

Java 线程的 6 种状态（NEW / RUNNABLE / BLOCKED / WAITING / TIMED_WAITING / TERMINATED）：

  • NEW：新建，线程被创建但未启动
  • RUNNABLE：运行中（包含 Ready 和 Running 两种状态，由系统调度）
  • BLOCKED：阻塞，等待获取监视器锁（synchronized 阻塞）
  • WAITING：无限等待（Object.wait()、LockSupport.park()、thread.join() 无参数）
  • TIMED_WAITING：有限期等待（Thread.sleep()、wait(timeout)、LockSupport.parkNanos()）
  • TERMINATED：终止，线程执行完毕
Q5
volatile 的作用是什么？它的局限性在哪里？
▼

volatile 的两大作用：

  • 保证可见性（Visibility）：当一个线程修改了 volatile 变量的值，其他线程能立即看到这个修改（通过 MESI 缓存一致性协议或总线锁实现）。不加 volatile 时，线程可能从 CPU 缓存或寄存器读取过期值。
  • 禁止指令重排序（Ordering）：volatile 在编译器和 CPU 之间插入内存屏障（Memory Barrier），防止编译器/CPU 对 volatile 变量前后的指令进行重排序。

volatile 不保证原子性：

volatile 的读/写本身是原子的（单次操作），但复合操作（如 i++ 实际上是"读取-修改-写入"三步操作）不是原子的，多线程下仍然会出现竞态条件。

例如：两个线程同时执行 i++，初始 i=0，期望结果 2，但实际可能只有 1：

线程A读取i=0 → 线程B读取i=0 → 线程A计算i+1=1 → 线程B计算i+1=1 → 线程A写入i=1 → 线程B写入i=1

volatile 的局限性：

  • 不保证原子性：复合操作需要用 synchronized 或 AtomicInteger 等原子类
  • 不保证互斥：无法阻止两个线程同时进入临界区

volatile 的典型使用场景：

  • 状态标志（volatile boolean running = true，用于优雅停机）
  • DCL（双重检查锁）中修饰单例对象的引用
  • 单写入多读取的场景（如配置信息读取）

Q1
MySQL 的锁机制有哪些？什么是间隙锁？
▼

按锁粒度分类：

  • 表锁：锁定整张表，开销小、加锁快、锁粒度粗。MyISAM 使用表锁；InnoDB 也支持表锁（LOCK TABLES）。
  • 行锁：只锁定需要修改的行，开销大、加锁慢、锁粒度细。InnoDB 的行锁基于索引实现（如果查询不走索引则升级为表锁）。
  • 页面锁：介于表锁和行锁之间，开销和粒度折中（BDB 存储引擎支持）。

按类型分类：

  • 共享锁（S Lock）：允许事务读取一行数据，多个 S 锁可同时加在同一数据上，不互斥。
  • 排他锁（X Lock）：允许事务删除/更新一行数据，X 锁与其他任何锁互斥（不能同时存在）。

InnoDB 特有锁：

  • 记录锁（Record Lock）：锁定索引记录（而非数据行），即最常见的行锁。
  • 间隙锁（Gap Lock）：锁定索引记录之间的间隙（或第一条/最后一条索引记录之前/之后的范围），防止其他事务在间隙中插入数据，防止幻读。Gap Lock 是 InnoDB 在 RR（可重复读）隔离级别下防止幻读的核心手段。
  • 临键锁（Next-Key Lock）：记录锁 + 间隙锁的组合。锁定一个前开后闭区间(prev, current]，既锁定已有的行，也锁定行之间的间隙。
  • 插入意图锁（Insert Intention Lock）：插入操作在等待获取间隙锁时，在间隙中设置的锁，表示有多个插入意图。

间隙锁举例：

id 列有值 1, 5, 10。当事务执行 SELECT * FROM t WHERE id > 3 AND id < 10 FOR UPDATE 时，Gap Lock 会锁定 (5, 10) 这个间隙，其他事务无法在这个范围内插入 id=6/7/8/9 的记录。

Q2
什么是慢查询？如何优化 SQL？
▼

慢查询定义：执行时间超过 slow_query_log 阈值（默认 1 秒）的 SQL 查询。

慢查询分析流程：

  • 开启慢查询日志：slow_query_log=1、long_query_time=1
  • 通过 mysqldumpslow 或 pt-query-digest 工具分析慢日志，找出 Top N 慢 SQL
  • 使用 EXPLAIN / EXPLAIN ANALYZE 分析执行计划，关注 type、key、rows、Extra
  • 使用 SHOW FULL PROCESSLIST 查看当前连接和查询状态

SQL 优化策略：

  • 索引优化：为 WHERE、JOIN、ORDER BY、GROUP BY 字段建索引；避免全表扫描（type=ALL）；使用覆盖索引减少回表
  • 避免索引失效：不在索引列上使用函数（WHERE YEAR(create_time)）、避免类型转换、遵循最左前缀原则
  • 优化数据访问：避免 SELECT *；分页优化（延迟关联 / 游标分页）；使用 LIMIT 限制返回行数
  • 减少 JOIN：大表 JOIN 消耗巨大，禁止超过 3 张表的 JOIN；可适当反范式化（冗余字段）
  • 避免排序：无法利用索引的 ORDER BY 会产生 filesort（内存/磁盘排序），可优化为索引排序
  • 批量操作：批量 INSERT 代替循环单条插入；批量 UPDATE 用 CASE WHEN 合并
  • 架构层优化：读写分离；引入缓存；分库分表
Q3
什么是分库分表？ShardingSphere 的实现原理是什么？
▼

分库分表（Sharding）的背景：

单机数据库的连接数、存储容量、写入性能都有上限。当单表数据量超过千万级（MySQL 行锁竞争加剧、B+ 树层数增加）时，需要将数据分散到多个库/表中。

分库：将数据按业务模块拆分到不同数据库实例，减轻单一数据库的 I/O 和连接压力。

分表（水平分表 / 垂直分表）：

  • 垂直分表：按字段的访问频率和相关性拆分（冷热字段分离），如将大字段（text/blob）拆分到扩展表，减少单行数据宽度。
  • 水平分表：按某个字段（分片键/sharding key）将数据行分散到多个表中（如按 user_id 取模、按时间范围分表）。

分片算法：

  • 哈希分片：sharding_key % N，数据分布均匀，但扩缩容时需要迁移大量数据（改 N 后所有路由变化）
  • 范围分片：按时间/ID 区间分，数据分布有序，适合范围查询，但易产生热点
  • 一致性哈希：解决扩缩容迁移量大的问题，Redis Cluster / ShardingSphereJDBC 都支持

ShardingSphere 的两种模式：

  • ShardingSphere-JDBC（分库分表客户端）：在应用层通过 jar 包拦截 SQL，重写 SQL 并根据分片键路由到对应库/表。轻量、无额外组件，但应用需要引入 SDK。
  • ShardingSphere-Proxy（分库分表代理端）：部署一个 MySQL 代理进程，应用连接代理而非直连 MySQL，代理负责 SQL 改写和路由。应用无侵入，但多了一层网络转发。

分库分表的问题：跨库查询困难（需多次查询或 ES）、分布式 ID 生成问题、分布式事务问题。

Q4
MySQL 的执行计划（EXPLAIN）需要关注哪些字段？
▼

使用 EXPLAIN [ANALYZE] <SQL> 分析 SQL 的执行计划。

关键字段：

  • type（连接类型，从优到差）：const > eq_ref > ref > range > index > ALL
    • const：主键或唯一索引等值查询，最多返回一行（最优）
    • eq_ref：关联查询时，被驱动表使用唯一索引（主键/唯一）访问（优）
    • ref：非唯一索引等值查询（良）
    • range：索引范围扫描（BETWEEN、IN、> 等）（中）
    • index：全索引扫描（只需要扫描索引，不需要回表）（差）
    • ALL：全表扫描（最差，需要优化）
  • key：实际使用的索引名称
  • key_len：索引使用的字节数（越长说明索引覆盖越宽）
  • rows：估算需要扫描的行数（越小越好）
  • Extra（重要）：
    • Using index：覆盖索引，无需回表（性能好）
    • Using index condition：索引下推（Index Condition Pushdown），在索引层过滤部分条件
    • Using where：需要在 MySQL 服务层额外过滤数据
    • Using filesort：无法利用索引排序，需要额外排序（可能使用磁盘或内存排序），需优化
    • Using temporary：需要创建临时表存储中间结果，需优化
    • Using join buffer：JOIN 时使用 join buffer 加速（Block Nested-Loop Join）
Q5
MySQL 死锁是怎么产生的？如何排查和解决？
▼

死锁产生的条件（四个必要条件，缺一不可）：

  • 互斥：资源一次只能被一个事务持有
  • 持有并等待：事务在持有资源的同时请求其他资源
  • 不可抢占：已持有的资源不能被强制释放，只能主动释放
  • 循环等待：事务间形成资源循环等待链

MySQL InnoDB 死锁典型场景：

两个事务以不同顺序操作相同行的记录：

事务A: UPDATE t SET v=1 WHERE id=1（锁住 id=1） → UPDATE t SET v=1 WHERE id=2（等待 id=2）

事务B: UPDATE t SET v=1 WHERE id=2（锁住 id=2） → UPDATE t SET v=1 WHERE id=1（等待 id=1）

循环等待，死锁形成。

死锁的检测与处理：

InnoDB 使用等待图（Wait-For Graph）算法自动检测死锁（O(N) 时间复杂度）。检测到死锁后，选择undo log 最小的事务进行回滚（通常是插入/更新数据量最少的事务），让其他事务继续执行。

死锁排查方法：

  • SHOW ENGINE INNODB STATUS：查看最近一次死锁的详细信息（包含两个事务的完整 SQL 序列、等待的锁信息）
  • 开启死锁日志：innodb_print_all_deadlocks=1（MySQL 5.6.2+）写入错误日志
  • 分析慢查询日志，找出高并发下频繁操作同一批记录的事务

避免死锁的措施：

  • 按固定顺序访问数据（所有事务都按 id 升序访问）
  • 减少事务持有锁的时间（尽快提交/回滚）
  • 降低隔离级别（如从 RR 降到 RC，Gap Lock 会减少）
  • 批量操作尽量一次锁定范围更大的记录

Linux 与容器 6 题
Q1
Linux 的 IO 模型有哪些？同步与异步、阻塞与非阻塞的区别？
▼

5 种 IO 模型（以 recvfrom 为例）：

  • 阻塞 IO（BIO）：应用进程发起 recvfrom 调用后，进程阻塞，直到数据复制完成才返回。简单但效率低，一个连接一个线程。
  • 非阻塞 IO（NIO）：应用进程发起 recvfrom 后，内核数据未就绪则立即返回 EWOULDBLOCK，进程不断轮询直到数据就绪，再进行数据复制并返回。CPU 空转开销大。
  • IO 多路复用（Multiplexing）：用一个线程管理多个 socket。通过 select/poll/epoll 监听多个 fd，当某个 socket 数据就绪后返回，再调用 recvfrom 读取数据。Linux 高并发服务器的基石。
  • 信号驱动 IO（SIGIO）：应用进程先注册 SIGIO 信号处理函数，内核数据就绪时发送 SIGIO 信号，进程收到信号后调用 recvfrom 读取数据。内核层面实现了异步通知。
  • 异步 IO（AIO / io_uring）：应用进程发起 aio_read 后立即返回，内核完成数据读取并复制到用户空间后，通过信号或回调通知进程。真正的异步，Linux 7 支持 io_uring。

同步 vs 异步：同步指用户进程发起 IO 后，需要等待（主动等待）结果才继续；异步指用户进程发起 IO 后立即返回，内核完成全部工作后通知进程（被动通知）。

阻塞 vs 非阻塞：阻塞指调用在数据未就绪时挂起；非阻塞指调用在数据未就绪时立即返回错误码，不挂起（但通常需要轮询）。

Q2
epoll 相比 select / poll 的优势是什么？
▼

select 的问题：

  • fd 数量限制：受限于 FD_SETSIZE（Linux 通常 1024），无法支持高并发
  • 线性扫描：每次调用都需要将 fd 集合从用户态复制到内核态，然后遍历所有 fd 检查就绪状态（O(N) 复杂度）
  • 重置成本：每次 select 调用后需要重置 fd 集合

poll 的改进：用 pollfd 数组代替 fd_set，突破了 fd 数量限制（仅受文件描述符上限 ulimit 限制），但仍然是线性扫描（O(N)）。

epoll 的优势：

  • 时间复杂度 O(1)：epoll 使用红黑树管理 fd，新增/删除 fd 和事件都在 O(log N)；就绪时只返回已就绪的 fd（通过回调机制），无需遍历所有 fd。
  • 无需重复拷贝：fd 集合在 epoll_ctl 时一次性从用户态拷贝到内核态，epoll_wait 只拷贝已就绪的 fd 列表，不需要每次都全量拷贝。
  • 支持边缘触发（ET）和水平触发（LT）：ET 模式只通知一次（需一次性读完所有数据），减少系统调用次数；LT 模式只要数据未读完就持续通知。

epoll 的应用：Nginx、Redis、Java NIO（Selector 底层实现）均使用 epoll，是 Linux 高性能网络 IO 的核心。

Q3
Docker 的原理是什么？容器与虚拟机的区别是什么？
▼

Docker 核心原理：

  • Linux Namespace（命名空间）：隔离资源，使容器内的进程看到不同的视图。6 种命名空间：
    • PID Namespace：隔离进程 ID（容器内 PID 1 可能对应宿主机 PID 1000）
    • Network Namespace：隔离网络设备、端口、路由表
    • Mount Namespace：隔离文件系统挂载点
    • UTS Namespace：隔离主机名和域名
    • IPC Namespace：隔离信号量、共享内存、消息队列
    • User Namespace：隔离用户和用户组 ID
  • Linux Cgroup（Control Groups）：限制和隔离进程的资源使用（CPU、内存、IO、网络带宽）。每个容器有自己的 cgroup 层级，防止一个容器耗尽宿主机资源。
  • UnionFS（联合文件系统）：Docker 镜像由多层只读层叠而成（Base Image + 中间层），容器启动时在最上层添加一个可写层（Container Layer）。容器间共享底层只读层，复用存储（Copy-on-Write）。

容器 vs 虚拟机：

  • 架构：VM 需要独立内核，通过 Hypervisor 模拟硬件，运行完整 OS（Guest OS），每个 VM 都有独立的 OS（数 GB）；容器共享宿主机内核，轻量得多（MB 级），启动快（秒级 vs 分钟级）
  • 隔离性：VM 隔离性更强（硬件级别隔离）；容器是内核级隔离，安全性相对弱（共用内核是潜在攻击面）
  • 性能：容器几乎无性能损失（接近原生）；VM 有虚拟化开销（CPU/IO/网络）
  • 启动速度：容器秒级启动；VM 分钟级启动
  • 适用场景：VM 适合需要强隔离的场景（多租户、不同 OS）；容器适合微服务、CI/CD、快速弹性伸缩

Q6
软链接和硬链接的区别是什么？
▼

硬链接（Hard Link）：

多个文件名指向同一个 inode（索引节点）。删除任意一个文件名，只要还有链接指向该 inode，数据就不会被删除。只有在同一文件系统内才能创建硬链接，不能跨文件系统，不能给目录创建硬链接。

软链接（Symbolic Link / Symlink）：

创建一个特殊类型的文件，内容是目标文件的路径（快捷方式）。软链接有自己的 inode，指向目标文件的路径。可以跨文件系统，可以给目录创建软链接。如果目标文件被删除（inode 链接数归 0），软链接变成"悬空链接"（ dangling link），访问会报错。

对比：

  • 跨文件系统：软链接可以，硬链接不行
  • 指向目录：软链接可以给目录创建；硬链接不行
  • 原文件删除后：硬链接仍然有效（因为引用数 > 1）；软链接变成悬空链接
  • 速度：硬链接直接指向 inode，快；软链接需要二次查找，慢
  • 创建命令：ln source hardlink（硬链接）；ln -s source symlink（软链接）

应用场景：硬链接用于安全删除（防止误删重要文件）；软链接用于程序版本切换（如 /usr/bin/python → /usr/bin/python3.9）、目录快捷方式、跨文件系统引用。

Q1
如何设计一个秒杀系统？
▼

秒杀系统的核心挑战：高并发读 + 写放大（万级 QPS 集中写入），必须解决缓存穿透、缓存击穿、数据库压力等问题。

整体架构（多层防护）：

第一层：客户端优化：

  • 验证码/答题，错开下单时间（前端加倒计时，服务端校验时间窗口）
  • 禁止重复点击（前端按钮置灰 + 服务端幂等校验）
  • CDN 加速静态资源

第二层：接入层（限流 + 防刷）：

  • API 网关限流（令牌桶 / 滑动窗口，限制 QPS）
  • IP 黑名单、User ID 限流（防爬虫/黄牛）
  • 热点资源 CDN 缓存（活动页面静态化）

第三层：应用层（缓存 + 异步）：

  • 秒杀活动开始前，商品信息预热到 Redis（库存数量、秒杀价格）
  • 库存预扣减：DECR stock 原子扣库存，返回负数则说明已售罄，直接返回失败（不用打到数据库）
  • 扣减成功则写入秒杀订单消息队列（Kafka / RabbitMQ）
  • 消息队列消费者异步处理订单创建、支付、库存最终一致性

第四层：数据层：

  • 数据库乐观锁更新库存：UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0
  • 库存分桶：将单 SKU 库存分成多个桶（1000件分10桶，每桶100件）写入不同 Redis key，分散并发压力
  • 订单库分库分表（按用户ID分库，按订单ID分表）

其他关键点：前端页面的库存展示需要定时从 Redis 更新；秒杀结束后及时更新库存回源；做好监控告警（Redis 连接数、MQ 积压量、数据库 QPS）。

Q2
如何设计一个短链接（短网址）系统？
▼

核心需求：长 URL → 短码（如 6 位字符串）→ 302 重定向到原 URL。

短码生成方案：

  • 哈希算法：对长 URL 做 MD5/SHA256，取前 6 位。缺点：哈希碰撞需处理；无法保证同一 URL 生成相同短码（去重困难）。
  • 自增 ID（分布式 ID）：为每个 URL 分配一个全局自增 ID，将 ID 转换为 62 进制（0-9a-zA-Z，每位62种可能）。6 位可表示 62^6 ≈ 568 亿。同一 URL 每次生成相同短码（去重查询）。
  • 预分配池：提前生成一批短码存入数据库，应用直接取用，避免自增 ID 的单点问题。

存储设计：

  • Redis 缓存：SET short_code original_url EX 86400（TTL 如 1 年），热点链接走缓存，O(1) 查找
  • MySQL 持久化：表设计：(id, short_code, original_url, created_at, expire_at, click_count)，对 short_code 建唯一索引
  • 存储容量预估：6 位短码 ≈ 568 亿可用，存储压力不大

302 vs 301：使用 302 临时重定向（保留统计价值）；301 会被浏览器缓存，导致无法统计。

高并发优化：Redis 缓存热点数据；CDN 节点预缓存高频短链；多级缓存（L1 本地 + L2 Redis + L3 MySQL）。

Q3
如何设计一个 Feed 流系统（朋友圈 / 微博）？
▼

Feed 流系统分为两种模式：Push（推）和Pull（拉），各有优劣。

Push 模式（写入时推送给粉丝）：

用户发微博时，将内容写入自己的发件箱（outbox），同时异步推送到所有粉丝的收件箱（inbox）。读取时直接从收件箱拉取。优点：读取极快（O(1)）；缺点：粉丝多时写放大严重（明星用户发一条微博要推送给千万粉丝）。

Pull 模式（读取时拉取关注列表）：

用户读取 Feed 时，查询自己关注的所有用户列表，再从每个关注用户的发件箱中拉取最新 N 条内容，然后归并排序。优点：写操作轻量；缺点：读取慢（需要多次查询 + 归并排序），读放大。

混合方案（业界常用）：

  • 普通用户：采用 Push（推送到粉丝 inbox）
  • 大 V 用户：采用 Pull（不主动推送，读取时拉取大 V 的发件箱）或 Hybrid（活跃粉丝推送，大 V 拉取）
  • Timeline Service：读取时从收件箱拉取 N 条（分页加载），优先读 Redis 缓存（收件箱列表），缓存 miss 则回源 MySQL

存储选型：

  • 发件箱/内容表：MySQL + 分库分表（按用户 ID），存储微博正文（可水平分表，按时间或 user_id 分）
  • 收件箱（Feed 列表）：Redis ZSet（key = user:inbox:{uid}，score = timestamp，member = feed_id），支持按时间倒序分页
  • 内容缓存：Redis String / Hash 存储微博正文（feed_id → content），热点内容走缓存

Feed 推拉平衡的权衡：Push 适合粉丝量少（KOL 以下）、更新频繁的应用；Pull 适合粉丝量差异大（微博类）或有大量大 V 的场景。

Q4
如何设计一个抢红包系统？
▼

核心挑战：高并发下的资金安全（不能超发红包）、原子性、公平性（随机金额）。

发红包：

  • 用户请求 → 账户余额扣款（事务保证原子性）
  • 生成红包唯一 ID（分布式 ID），计算每人金额（二倍均值法：每次随机 [0, 剩余金额*2/剩余人数]），将金额和状态存入数据库 + Redis（ZSet，score=红包ID）
  • 红包金额明细预拆（生成每个子红包的金额和领取状态），避免抢的时候再算金额

抢红包：

  • Redis 原子扣减：DECR redpack:{id}:remain_count，返回负数说明已抢完（防超卖，不需要数据库锁）
  • 从预拆明细中随机选一个未领取的子红包（可用 SET 存储未领取 ID，随机弹出一个 SPOP）
  • 更新子红包状态为"已领取"，更新领取者、领取时间
  • 异步发送领取成功通知（MQ）

防并发安全：

  • 乐观锁 + 版本号：更新子红包时 WHERE id=? AND status=0，影响行数为 0 说明已被抢
  • Redis Lua 脚本：将"查询+扣减+获取子红包"打包为原子操作，避免并发竞争

超时未领退款：定时任务扫描 24 小时未领取的红包，将金额退回原账户。

Q5
如何设计一个延迟消息队列？
▼

延迟队列（Delay Queue / Delayed Message）指消息在发送后不立即投递，而是等待指定时间后才投递给消费者。常见场景：订单超时未支付自动取消、会员到期提醒、异步任务调度。

实现方案：

1. RocketMQ 延迟消息（原生支持）：

设置消息的 delayTimeLevel（延迟级别，如 1s/5s/30s/2m...），消息先写入延迟主题（SCHEDULE_TOPIC），RocketMQ 的定时任务根据延迟级别将消息投递到真实消费队列。优点：原生支持；缺点：延迟级别固定（需配置，不支持任意精确延迟）。

2. RabbitMQ + 死信队列（DLX） + TTL：

为队列设置 TTL（消息过期时间），消息 TTL 过期后进入死信队列（DLX），消费者监听 DLX 获取延迟消息。缺点：TTL 单条消息设置不灵活（每个消息的延迟时间不同需要单独队列）；消息顺序性可能改变。

3. Redis ZSet 延迟队列（轻量级）：

将消息存入 Redis ZSet，score = 执行时间戳（Unix timestamp），用一个定时任务（Lua 脚本定时轮询）不断读取 score < current_time 的消息进行消费。优点：实现简单，适合延迟精度要求不高（秒级）、消息量不大的场景。

4. 时间轮算法（Netty / Kafka Timer）：

用时间轮（TimeWheel）数据结构管理延迟任务，每个时间格维护一个 bucket（链表），指针周期性推进（Tick），到期任务自动触发。时间复杂度 O(1)，适合高性能定时器（Netty HashedWheelTimer、Kafka client.timer）。

5. 分布式延迟队列（如当当网 elastic-job、xxl-job）：

使用 Quartz /xxl-job 等分布式任务调度框架，将延迟任务作为定时 Job 注册，执行时查询数据库/Redis 获取到期的任务。

Q6
如何设计一个 IM（即时通讯）系统？
▼

IM 系统的核心问题：消息可靠投递（不丢失、不重复）、实时性、消息顺序性、高并发。

消息通道：

  • WebSocket（双向）：客户端与服务器建立持久连接，服务端可主动推送。是 IM 实时通信的基础。
  • 轮询（HTTP Short Polling）：客户端定时请求拉取消息，简单但效率低、延迟大。
  • 长轮询（Comet）：客户端请求后服务端 hold 住连接直到有新消息或超时，减少无效请求。

消息路由架构：

  • 消息 ID（MsgID）：全局唯一 ID（雪花算法），用于去重和排序。
  • 写扩散 vs 读扩散：
    • 写扩散（Push）：发消息时主动推送到接收者的收件箱（类似 Feed 流 Push），读取快（直接读 inbox），写入量大（每个在线用户都要写一次）。
    • 读扩散（Pull）：发消息只写发件箱，读取时拉取关注列表再聚合。写入轻量，读取重。
    • 生产级 IM（如微信）通常混合使用：小群写扩散、大群读扩散。

消息可靠投递：

  • 消息确认（ACK）：客户端收到消息后发送 ACK，服务端收到 ACK 才标记已读。超时未 ACK 则重发。
  • 消息存储：离线消息写入消息库（MySQL + 分库分表 / MongoDB），用户上线后拉取离线消息。
  • 消息漫游：所有消息持久化，支持任意时间跨设备拉取历史记录。

在线状态：

使用 Redis Set 存储用户的在线状态（uid → Gateway Server IP）。用户上线时加入 Set，下线时移除（通过心跳检测超时移除）。查询用户在线状态时读 Redis 即可。
