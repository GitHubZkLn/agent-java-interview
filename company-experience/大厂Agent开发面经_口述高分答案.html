Agent 核心问题

大模型 Agent 开发核心面试题

Q1 在 Agent 知识闭环中，如何设计决定哪些信息进入向量数据库（长期记忆）、哪些进入上下文窗口（短期记忆）、哪些直接转化为模型权重的元记忆？
口述高分答案

这是一个关于分层记忆架构设计的经典问题。

我的设计思路是采用三层记忆架构：

1. 上下文窗口（工作记忆）：只存放当前对话的上下文、当前任务的中间状态、以及通过 Attention Sink 锚定的关键信息。一般限制在 4K-8K tokens。

2. 向量数据库（长期记忆）：存放经过提炼的事实性知识、用户画像、历史交互模式、成功的工具调用经验等。容量可达百万级。

3. 模型权重（内化知识）：通过微调将高频、通用、稳定的知识固化到模型权重中，如通用推理模式、常用工具调用规范等。

核心判断标准：

• 频次：出现 3 次以上的模式 → 进入长期记忆

• 时效性：超过 7 天未使用 → 从工作记忆降级

• 通用性：跨用户适用的知识 → 考虑权重化

• 事实性：需要精确回忆的 → 长期记忆；需要即时推理的 → 工作记忆

来源：互联网大厂面经
Q2 当对话轮数较多且上下文窗口严重不足时，如何在不丢失初始 Attention Sink 的前提下保持生成的连贯性？
口述高分答案

这是一个工程实践中的关键问题。我主要采用以下策略：

1. Attention Sink 锚定机制：在对话开始时明确设置关键实体（如用户名、项目名）的向量表示，在压缩时始终保留。

2. 增量压缩而非全量丢弃：使用summarization + selective retention：对历史对话做摘要，保留关键实体和决策节点的完整信息。

3. 分层压缩策略：将对话分为"核心事实层"（始终保留）、"推理过程层"（摘要保留）、"中间结果层"（可丢弃）。

4. 语义压缩：用小模型对历史对话做语义压缩，提取核心意图和关键信息。

5. 层级索引：建立关键信息的索引表，压缩后仍能快速检索。

来源：互联网大厂面经
Q3 详细讲讲你设计的 Agent 是如何实现的？在"推理-行动"循环中，如何设计来纠正逻辑塌缩或无效工具调用？
口述高分答案

我的 Agent 系统采用 ReAct + Self-Correction 架构：

核心循环：Thought → Action → Observation → Reflection → Decision

防止逻辑塌缩的机制：

1. 步数熔断器：设置最大推理步数（如 15 步），超限则强制输出当前最优解。

2. 重复检测：维护已执行动作的 hash 表，连续 3 次相似动作触发告警。

3. 状态验证：每个 action 后验证状态变化是否符合预期，不符合则回退。

4. 置信度阈值：工具调用置信度低于 0.7 时，要求人工确认或切换策略。

5. 反思机制：每次推理后让模型评估"这个推理是否合理"，有问题则回退。

来源：字节 Agent 开发面经
Q4 在电商或导购场景下，用户的请求往往高度模糊，Agent 如何精准理解这种需求？
口述高分答案

电商场景的模糊需求处理是核心能力，我的方案：

1. 主动澄清策略：

• 预设澄清意图识别：当 query 包含模糊词（"好看的"、"合适的"）时触发

• 区分"反问"vs"强行推断"：高风险决策（如购买）反问，低风险（如浏览）可推断

2. 历史画像辅助：

• 结合用户历史行为（浏览、收藏、购买）构建偏好向量

• 模糊词映射到历史偏好（如"显瘦"→ 用户常购版型）

3. 渐进式披露：

• 不一次性问完，先问最有区分度的问题

• 用最少问题定位需求（一般 2-3 个）

4. 置信度决策树：置信度 > 0.85 → 直接推荐；0.6-0.85 → 推荐 + 说明；< 0.6 → 反问

Q5 针对包含 3 个以上工具调用且高频请求的任务，通过什么方式可以压低系统整体的端到端延迟？
口述高分答案

降低 Agent 工具调用延迟的实战经验：

1. 并行化调用：

• 无依赖的工具调用并行执行（如同时查库存、查物流、查优惠）

• 使用 asyncio + batch API

2. 工具预热：

• 高频工具保持长连接

• 热点数据缓存（如商品信息）

3. 预测性调用：

• 根据用户意图预测下一步工具，提前调用

• 构建工具调用概率模型

4. 轻量化路由：

• 用小模型做意图识别和工具路由，减少 LLM 调用

• 规则 + 小模型混合策略

5. 异步化处理：非关键路径工具异步调用，主流程不等待

Q6 如何衡量 Agent 的 Planning 能力 vs Hallucination Rate？请列举具体的量化评估指标或自动化评估框架。
口述高分答案

Planning 能力评估指标：

1. 任务完成率：Agent 能否独立完成给定任务

2. 路径最优性：实际执行路径 vs 最优路径的比值

3. 规划步数准确率：预测步数 vs 实际步数的偏差

4. 工具调用准确率：正确选择工具的比例

Hallucination Rate 评估：

1. 事实错误率：输出中与事实不符的比例

2. 自我一致性得分：同一问题多次回答的一致性

3. 引用准确性：引用内容与原文的匹配度

自动化评估框架：

• G-Eval：利用 LLM 做自动化评估

• RAGAS：专门评估 RAG 系统的幻觉率

• Self-Evaluation：Agent 自我评估置信度

Q7 Token 过长导致的 Attention 稀释现象为什么会导致 Agent 的指令遵循能力下降？
口述高分答案

Attention 稀释的原理与影响：

1. 计算资源分散：Attention 计算量 O(n²)，token 越长，每个 token 分到的注意力权重越平均

2. 关键信息被淹没：重要指令（如"不要做什么"）的注意力权重被稀释

3. 位置编码衰减：长序列下早期 token 的位置信息表达能力下降

4. 实验数据：研究表明，超过 32K tokens 时，指令遵循准确率下降约 30%

解决方案：

• Sparse Attention（只关注关键 token）

• Information Distillation（压缩冗余信息）

• Hierarchical Attention（分层处理）

来源：互联网大厂面经
Q8 面对模型在 Agent 执行过程中出现的循环调用或陷入思维死循环问题，有哪些解决方法？
口述高分答案

死循环是 Agent 落地的核心难题，我的解决方案：

1. 熔断机制：

• 步数限制：max_steps = f(任务复杂度)

• 时间限制：单次执行不超过 30s

• 成本限制：token 消耗超阈值则停止

2. 状态记忆检测：

• 维护历史状态的 hash，如果 N 步内状态重复 → 死循环

• 相似度检测：当前状态与历史状态相似度 > 0.9 → 触发

3. 决策多样性检测：

• 记录历史决策，如果连续 3 次决策相似 → 提示重新思考

4. 回退与重试：

• 检测到死循环时，回退到上一步，用更明确的 prompt 重试

5. 外部干预：

• 高风险场景强制人工确认

• 置信度低于阈值时暂停，等待指令

Agent 记忆系统设计

上下文、长期记忆、工作记忆设计

Q1 上下文窗口满了如果需要压缩，怎么去压缩？如何筛选有效信息和垃圾信息，保证正确率最大？
口述高分答案

上下文压缩是工程落地的关键问题。

压缩策略：

1. 重要性评分：用轻量级模型对每轮对话打分，保留高分内容

2. 实体保留：始终保留关键实体（人名、日期、关键决策）

3. 时间衰减：越近的对话权重越高，历史对话逐步降权

4. 意图聚类：将相似意图的对话合并，只保留核心摘要

垃圾信息识别：

• 重复信息：与已有内容相似度 > 0.8 → 丢弃

• 无效闲聊：不含关键实体的对话 → 可丢弃

• 错误纠正：被后续对话否定的信息 → 标记删除

保证正确率的方法：

• 保守压缩：只压缩置信度 > 0.9 的部分

• 双缓存：保留原始上下文快照，支持回溯

来源：字节 Agent 开发面经
Q2 会话记忆功能如何实现？一次性记忆和永久性记忆分别如何实现？
口述高分答案

三层记忆架构设计：

1. 一次性记忆（会话级）

• 存储位置：内存 + Redis Session

• 生命周期：会话结束即销毁

• 实现：简单的 KV 存储，key = session_id，value = 对话历史

2. 短期记忆（用户级）

• 存储位置：向量数据库（用户维度索引）

• 生命周期：30 天无交互后降级或删除

• 内容：用户偏好、近期任务、历史交互摘要

3. 永久性记忆（系统级）

• 存储位置：知识图谱 + 结构化数据库

• 内容：事实性知识、通用规则、成功案例

• 更新：需要显式触发，评估后写入

记忆流转：一次性 → 短期 → 永久（需要评估晋升）

Q3 Agent 中长短期记忆如何设计？各自存什么？什么时候触发读取？
口述高分答案

长短期记忆设计：

短期记忆（Working Memory）：

• 存储：当前对话上下文、当前任务状态、中间推理结果

• 触发：每次对话开始时自动加载

• 容量：限制 8K tokens，超出时压缩

长期记忆（Persistent Memory）：

• 存储：用户画像、历史偏好、跨会话知识、工具调用经验

• 触发：意图识别相关时检索

• 索引：按用户 ID + 实体类型复合索引

触发读取时机：

1. 会话开始：加载用户长期记忆，初始化上下文

2. 任务开始：检索相关历史任务经验

3. 工具调用前：加载相似任务的调用经验

4. 任务结束：评估是否需要写入长期记忆

来源：互联网大厂面经
Q4 多轮对话中，如果不同轮次的记忆发生冲突，如何处理？
口述高分答案

记忆冲突处理策略：

1. 时间戳优先级：最新信息覆盖旧信息（适用于用户主动更正）

2. 置信度评估：

• 明确声明 > 模糊暗示

• 具体数值 > 模糊描述

• 重复确认 > 单次提及

3. 来源可靠性：

• 外部知识库 > Agent 推断 > 用户闲聊

• 认证来源 > 普通来源

4. 冲突标记：

• 将冲突信息都保留，标记时间戳

• 输出时说明"根据某时间点的信息..."

5. 主动澄清：当冲突无法解决时，询问用户确认

Q5 你们上下文是怎么管理的？上下文是共享的吗？具体会共享哪些信息？
口述高分答案

上下文管理方案：

管理策略：

1. 会话级别隔离：每个用户会话独立的上下文

2. Agent 级别独立：每个 Agent 维护自己的上下文

隔离信息：

• Agent 私有推理过程

• 各 Agent 的中间状态

• 敏感用户信息

实现方式：Redis 存储会话上下文 + 消息队列同步共享信息

RAG 系统

检索增强生成核心面试题

Q1 RAG 这块，怎么判断召回"够用了"？有没有一个明确的评估标准？
口述高分答案

RAG 召回质量评估的实战经验：

定量指标：

1. Recall@K：Top-K 召回结果中包含正确答案的比例

2. MRR（Mean Reciprocal Rank）：正确答案平均排名的倒数

3. NDCG：考虑排序质量的召回指标

"够用"的标准：

• 简单问答场景：Recall@3 > 90%

• 复杂推理场景：Recall@10 > 85% + MRR > 0.7

• 严格事实场景：必须 Recall@1 命中

人工评估阈值：

• 标注 100 条样本，召回数满足率 > 85%

• 召回内容相关性平均分 > 4/5

来源：互联网大厂面经
Q2 向量检索你们是怎么评估效果的？Embedding 模型你们有做过 A/B 测试吗？指标是怎么设计的？
口述高分答案

向量检索效果评估：

离线指标：

• Recall@K：召回率

• MRR：排序质量

• Latency：P99 < 50ms

A/B 测试设计：

1. 分组策略：流量 50/50 划分，用户维度哈希分流

2. 核心指标：

• 任务完成率（是否找到答案）

• 用户满意度评分

• 平均对话轮次

3. 辅助指标：

• 首 token 延迟

• Token 消耗成本

4. 测试周期：至少 2 周，覆盖完整用户周期

5. 统计显著性：p-value < 0.05，置信度 > 95%

来源：互联网大厂面经
Q3 如果向量召回和关键词召回结果冲突，怎么决策？混合检索是并行召回还是分阶段？为什么这么设计？
口述高分答案

混合检索设计：

冲突决策策略：

1. 加权融合：Score = α × vector_score + (1-α) × BM25_score

2. RRF（Reciprocal Rank Fusion）：1/rank1 + 1/rank2 融合

3. 场景自适应：精确查询偏 BM25，语义查询偏向量

并行召回 vs 分阶段：

采用并行召回 + 结果融合：

• 优势：同时利用两种召回的优点

• 劣势：计算资源翻倍

• 适用场景：精确查询 + 语义相似查询

分阶段适用于：先粗筛（向量），再精排（关键词）

来源：互联网大厂面经
Q4 Query 改写如果改错了怎么办？有没有遇到"召回正确但生成胡说"的情况？你怎么处理？
口述高分答案

Query 改写错误的处理：

1. 回退策略：检测到改写异常时，使用原始 query 召回

2. 一致性校验：改写后 query 与原始 query 做语义相似度检测

3. 多版本保留：同时保留改写和原始 query 的召回结果

"召回正确但生成胡说"的问题：

这是 RAG 的核心难题！我的处理方案：

1. 严格引用溯源：强制模型引用召回内容，限制自由发挥

2. 置信度阈值：召回内容相关性 < 0.7 时，明确告知"知识库无相关信息"

3. 事实核查层：生成后用小模型核查关键事实点

4. 拒答机制：超出知识库范围的 query 主动拒答而非编造

来源：互联网大厂面经
Q5 长文档为什么一定要切 chunk 再做向量化？不切会有什么问题？Chunk 切分时为什么要有重叠区域？比例一般怎么确定？
口述高分答案

Chunk 切分的必要性：

1. 向量维度限制：模型有 max_tokens 限制，长文档无法整体向量化

2. 语义精度：短文本语义更精确，长文本会稀释核心信息

3. 召回粒度：小块召回更精准，大块会引入无关信息

不切的问题：

• 召回结果包含大量无关上下文

• 召回准确率下降

• 计算资源浪费

重叠区域设计：

• 目的：避免边界信息丢失（同一语义被切分）

• 比例：通常 10%-20% 重叠

• 策略：按句子切分 + 50-100 tokens 重叠

来源：互联网大厂面经
Q6 稠密向量和稀疏向量的区别是什么？各自适合什么场景？为什么需要 Rerank 模型？
口述高分答案

稠密向量 vs 稀疏向量：

稠密向量：

• 维度固定（如 768/1024）

• 语义相似度高

• 适合：语义搜索、近义词理解

稀疏向量：

• 高维（如 5万维），大部分为 0

• 精确关键词匹配

• 适合：精确术语搜索

Rerank 模型的作用：

向量检索追求召回速度，用近似最近邻（ANN），会有精度损失

Rerank 用Cross-Encoder 精排：

• 重新计算 query-doc 的精确相似度

• 考虑词项匹配、语义匹配的多维度

• 放在召回后，提升排序准确性

来源：互联网大厂面经
Q7 为什么在复杂的 Agent 闭环场景中，仅靠 RAG 无法彻底解决幻觉问题？模型幻觉在电商这种强事实性场景里怎么治理？
口述高分答案

RAG 的局限性：

1. 召回本身有损失，无法 100% 准确

2. 模型会过度泛化，即使有上下文也可能"自行发挥"

3. 知识库本身可能过时或不完整

4. 生成阶段仍有概率产生幻觉

电商场景幻觉治理方案：

1. 严格的事实核查：

• 价格、库存、活动规则 → 必须实时查询数据库

• 强事实性内容：模型只负责组织语言，不负责生成数据

Q1 Java 中 HashMap 的底层结构是怎样的？JDK 1.8 之后做了哪些改进？链表转红黑树的阈值是多少？为什么设置为这个值？
口述高分答案

HashMap 底层结构：

JDK 1.8 之前：数组 + 链表

JDK 1.8 及之后：数组 + 链表 + 红黑树

JDK 1.8 改进：

1. 引入红黑树，解决链表过长导致的查询退化 O(n) → O(logn)

2. 优化 hash 算法：(key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16)

3. 扩容机制优化：尾插法替代头插法，避免死循环

4. 容量为 2 的幂次方，优化取模运算

链表转红黑树阈值：

• 转换阈值：8

• 红黑树退链表阈值：6

• 设计原因：根据泊松分布，链表长度超过 8 的概率极低（0.00000006）

• 留 6 作为缓冲，避免频繁转换

3. 减少锁粒度

5. 避免在持有锁时调用外部方法

使用场景：

• CountDownLatch：主线程等待多个子任务完成

• CyclicBarrier：多线程互相等待，然后一起执行下一步

Q1 数据库的三范式是什么？LEFT JOIN 和 INNER JOIN 的区别是什么？为什么要使用 WHERE 条件替代 LEFT JOIN 的过滤？
口述高分答案

数据库三范式：

• 1NF：字段原子性，不可再分

• 2NF：满足1NF + 非主键字段完全依赖主键

• 3NF：满足2NF + 非主键字段不传递依赖主键

JOIN 区别：

• INNER JOIN：只保留两边都匹配的记录

• LEFT JOIN：保留左表全部记录，右表没有匹配的为 NULL

WHERE vs ON 过滤：

• ON 条件：决定是否连接（JOIN 阶段）

• WHERE 条件：决定是否保留（过滤阶段）

• 对于 LEFT JOIN，在 ON 中过滤右表会保留左表所有行；在 WHERE 中过滤会导致左表行被丢弃

最佳实践：

• 连接条件放在 ON

• 业务过滤放在 WHERE

高频面试算法题

Q1 手撕：第 K 大元素（要求时间复杂度 O(n)）
参考代码

快速选择算法：

基于快速排序的 partition 思想，平均 O(n)，最坏 O(n²)

思路：找到第 k 大的数，相当于找到排序后第 n-k 小的位置

快速选择模板 高频 
int findKthLargest(vector& nums, int k) {
    int left = 0, right = nums.size() - 1;
    while (true) {
        int pos = partition(nums, left, right);
        if (pos == k - 1) return nums[pos];
        else if (pos > k - 1) right = pos - 1;
        else left = pos + 1;
    }
}

int partition(vector& nums, int l, int r) {
    int pivot = nums[l];
    while (l < r) {
        while (l < r && nums[r] <= pivot) r--;
        nums[l] = nums[r];
        while (l < r && nums[l] > pivot) l++;
        nums[r] = nums[l];
    }
    nums[l] = pivot;
    return l;
}

Q3 手撕：螺旋矩阵
解题思路

模拟遍历

定义四个边界：上、下、左、右

按顺时针顺序遍历：右→下→左→上

每遍历完一条边，缩小对应边界

当边界相遇时停止

Q4 手撕：最短路径
解题思路

Dijkstra 算法（无负权边）

1. 使用优先队列维护最短距离

2. 每次取出未访问节点中距离最小的

3. 用该节点更新其邻居节点的距离

时间复杂度：O((V+E) log V)

Bellman-Ford 算法（可处理负权边）

对所有边进行 V-1 次松弛操作

Q5 简单说一下快速排序的实现逻辑和思想
口述高分答案

快速排序核心思想：分治法 + 原地分区

步骤：

1. 选择基准：通常选第一个或随机选

2. 分区（partition）：将数组分为两部分

- 左侧：小于等于基准

- 右侧：大于基准

3. 递归排序：对左右两部分递归执行

partition 实现：

• 双指针法：从两端向中间推进

• 挖坑法：基准挖出，两端交替填坑

复杂度：

• 平均：O(n log n)

• 最坏（有序数组）：O(n²)

• 空间：O(log n) 递归栈

优化：随机基准、三数取中、尾递归优化

来源：互联网大厂面经
